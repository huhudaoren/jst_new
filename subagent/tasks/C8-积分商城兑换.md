## 阶段 / 模块

## ⚠️ 实测 Schema 校正（派发前主 Agent 补录）

`jst_mall_goods` 实际列：
- 无 `category`、无 `role_limit_json`
- `goods_type VARCHAR(10)` (virtual/physical)、`status VARCHAR(10)` (draft/on/off)
- `role_limit VARCHAR(20)` (student/channel/both)
- `points_price BIGINT`、`cash_price DECIMAL(12,2)`
- `stock INT` + `stock_warning INT`（虚拟商品 stock 可 = -1 表示不限）

`jst_mall_exchange_order` 实际列：
- 主键 `exchange_id`（**不是** `order_id`）
- `exchange_no` / `user_id` / `goods_id` / `quantity` / `points_used` / `cash_amount`
- `order_id` 字段**存在**，指向现金部分的 `jst_order_main`（如有）
- `address_snapshot_json JSON`（实物商品用，入参不是 addressId，是 address 快照对象）
- `status VARCHAR(20)` / `logistics_company` / `logistics_no` / `ship_time` / `complete_time`
- `aftersale_status VARCHAR(20)`

`jst_points_account` 实际列：
- **不是 user_id**，是 `owner_type VARCHAR(10)` + `owner_id BIGINT`（多类型账户共表）
- 学生积分账户 `owner_type='user'`, `owner_id=userId`
- `available_points BIGINT` / `frozen_points BIGINT` / `version INT`（乐观锁已就位）
- C2 实现已有"冻结 → 扣减 → 解冻"模式，**直接复用** C2 OrderServiceImpl 的写法

`jst_user_coupon` 虚拟商品发放目标列：
- `coupon_template_id` (FK 指向 jst_coupon_template，C8 不操作 template 表)
- `user_id` / `issue_source VARCHAR(20)`（本任务写 `mall_exchange`）/ `issue_batch_no`
- `status VARCHAR(20)`（new/used/expired/voided）
- `valid_start` / `valid_end` / `order_id`（这里的 order_id 指 mall_exchange_order.exchange_id，与 jst_order_main 无关）

`jst_user_rights` 虚拟权益发放目标列：
- `rights_template_id` (FK，C8 不建 template)
- `owner_type='user'` + `owner_id=userId`
- `source_type='mall_exchange'` + `source_ref_id=exchangeId`
- `remain_quota` / `valid_start` / `valid_end` / `status`

**fixture 必须预置**至少 1 条 `jst_coupon_template` 和 1 条 `jst_rights_template`（或复用已有），否则发放时 FK 失败。

## 业务背景

PRD 7.5 / 8.7 积分商城：学生用积分（可叠加现金补差）兑换虚拟权益（优惠券/权益卡）或实物商品（周边/礼品）。
- 虚拟商品兑换即时到账（自动发券 / 写 user_rights）
- 实物走发货流程
- 售后退款按"现金→积分"顺序回滚，参考 C4 退款范式

## 状态机 SM-18

```
pending_pay → paid → pending_ship → shipped → completed
     ↓        ↓
 cancelled  aftersale → refunded / closed
```
虚拟商品省略 `pending_ship/shipped`，`paid → completed` 直接跳。

## 锁
- `lock:mall:goods:stock:{goodsId}` — 库存扣减（等 3s，持 5s）
- `lock:mall:exchange:{orderId}` — 状态推进互斥（等 2s，持 5s）
- `lock:points:freeze:{userId}` — 复用 C2 积分冻结锁

## 接口

### admin 侧
- GET/POST/PUT/DELETE `/jst/points/mall/goods` CRUD
- POST `/jst/points/mall/goods/{id}/publish` 上架
- POST `/jst/points/mall/goods/{id}/offline` 下架
- GET `/jst/points/mall/exchange/list` 所有兑换订单分页
- POST `/jst/points/mall/exchange/{orderId}/ship` 填快递单号

### wx 侧
- GET `/jst/wx/mall/goods/list?category=&pageNum=` 商品列表
- GET `/jst/wx/mall/goods/{id}` 详情
- POST `/jst/wx/mall/exchange/apply` ⭐ 兑换下单
  - 入参 `{ goodsId, quantity, addressId?, payMethod }`
  - 逻辑（@Transactional）：
    1. 锁 `lock:mall:goods:stock:{goodsId}`
    2. 查商品 FOR UPDATE，校验 status=on、stock 足够、用户角色符合 role_limit
    3. 计算 `totalPoints = price_points × qty`、`totalCash = price_cash × qty`
    4. 锁 `lock:points:freeze:{userId}`
    5. 乐观锁冻结积分：`UPDATE jst_points_account SET frozen_points = frozen_points + totalPoints, available_points = available_points - totalPoints, version=version+1 WHERE user_id=? AND version=? AND available_points >= totalPoints`
    6. 写 points_ledger 一行 `type='frozen'`
    7. INSERT `jst_mall_exchange_order` status=`pending_pay`（纯积分则直接 `paid` 触发发放）
    8. 若 totalCash > 0：调 `WxPayService` mock 路径；本期 wx 前端后续调 mock-success 完成
    9. 若 totalCash == 0：直接内部推进 `paid` → 若虚拟商品直接 `completed`（发券/权益）；若实物则 `pending_ship`
    10. 扣减 `jst_mall_goods.stock`
    11. 返回 ResVO
- POST `/jst/wx/mall/exchange/{orderId}/pay/mock-success` 积分+现金的现金部分支付成功回调（复用 C2 通道）
  - 推进状态 `pending_pay → paid`，并根据商品类型决定 `paid → completed`（虚拟即发）或 `paid → pending_ship`（实物）
  - 虚拟商品发放：INSERT user_coupon / user_rights（跨模块 lookup）
  - 积分从 frozen → deducted（points_ledger 写 `consumed`）
- POST `/jst/wx/mall/exchange/{orderId}/cancel`
  - pending_pay 可取消，积分从 frozen 解冻回 available，恢复库存
- GET `/jst/wx/mall/exchange/my` 我的兑换订单列表
- GET `/jst/wx/mall/exchange/{id}` 详情
