## 防雷点 (10 条强制清单)

### 1. 金额计算源不可信
前端**只能传** `couponId` + `pointsToUse`,**后端重算** list_amount/coupon_amount/points_deduct/net_pay/platform_bear/service_fee。前端传的金额仅用于展示对比,后端如发现不一致直接抛 AMOUNT_MISMATCH。

### 2. 一个大事务 (核心)

`OrderService.createMixedPay()` 必须 `@Transactional(rollbackFor = Exception.class)`,内部:
1. 校验 enroll (status=approved + order_id is null)
2. 校验 coupon (status=unused + 适用 + 未过期)
3. 校验 points (available_points >= pointsToUse)
4. 计算金额 (重算!)
5. 校验渠道方 (取 active binding 的 channel_id)
6. INSERT order_main + order_item (按 enroll 数量,本期通常 1 条)
7. UPDATE coupon → locked
8. UPDATE points_account: frozen += pointsToUse, available -= pointsToUse + 写流水
9. UPDATE enroll_record.order_id = newOrderId
10. INSERT rebate_ledger 状态 pending_accrual,按 order_item.rebate_share 写
11. 调微信支付 (Mock 模式直接返回假 prepayId)
12. 状态机 created → pending_pay
13. 返回 OrderResVO + WechatPrepayVO

任何一步失败,整个事务回滚 (积分不冻 / 券不锁 / 订单不存)。

### 3. 状态机校验
**所有**状态变更必须先 `OrderStatus.fromDb(...).assertCanTransitTo(target)`,否则抛 ILLEGAL_TRANSIT。

### 4. 乐观锁更新主状态
```sql
UPDATE jst_order_main SET order_status='paid', pay_time=NOW()
WHERE order_id = ? AND order_status = 'pending_pay'
-- updated == 0 抛 OPTIMISTIC_LOCK
```

### 5. 积分冻结/解冻必须用乐观锁
点数账户可能被 admin 调整 / 同时下多单,必须基于 version 字段或 frozen+available 的组合校验。

### 6. 优惠券锁定的并发安全
```sql
UPDATE jst_user_coupon SET status='locked', order_id=?
WHERE user_coupon_id = ? AND status = 'unused' AND user_id = ?
-- updated == 0 抛 COUPON_INVALID
```

### 7. 报名 → 订单 1:1 强约束
- enroll_record.order_id is null → 允许创建新订单
- enroll_record.order_id 已存在但订单 status=cancelled → 允许重新创建 (清旧 order_id)
- enroll_record.order_id 已存在且订单 status in (pending_pay, paid, ...) → 抛 DUPLICATE_ORDER

### 8. 零元订单特殊处理
list_amount = 0 (赛事 free 或 100% 抵扣):
- 跳过支付环节
- 直接 created → completed (不经 pending_pay)
- 不调微信支付
- 不写 payment_record
- coupon/points 仍然 locked → used (即使不付钱也消耗)
- order_type = 'zero_price'

### 9. 微信支付回调防重
回调接口加锁 + 校验 outTradeNo 是否已处理:
```java
String lockKey = "lock:wxpay:notify:" + outTradeNo;
return jstLockTemplate.execute(lockKey, 5, 10, () -> {
    OrderMain order = ...;
    if ("paid".equals(order.getOrderStatus())) {
        return SUCCESS; // 幂等返回
    }
    // 否则推进状态机
});
```

### 10. Mock 模式联调路径
开发期 (jst.wxpay.enabled=false):
- POST /jst/wx/order/create → 返回 `{orderId, fakePrepayId: "MOCK_xxx", actualPayAmount}`
- 前端拿到 fakePrepayId 后,**自动**调 POST `/jst/wx/pay/mock-success?orderId=xxx` (新增的 dev-only 接口)
- mock-success 接口直接走 wxpay notify 流程
- 此接口在 prod 环境**禁用** (用 @ConditionalOnProperty)

## 金额计算公式 (强制按此实现)

```
listAmount = contest.price * quantity (本期 quantity = 1)

couponAmount = 按 coupon_template 规则:
  - full_reduce: face_value (如 listAmount >= threshold)
  - direct_reduce: face_value
  - discount: listAmount * (1 - discount_rate)
  - contest_specific: 同上但限赛事

pointsCashRate = 100 (来自 application.yml jst.biz.points-cash-rate)
pointsDeductAmount = pointsToUse / pointsCashRate
  // 例: 1000 积分 = 10 元

platformBearAmount = 0  // 本期所有优惠由商家承担,不让平台 bear
                        // 后期 PRD 可能改为 coupon 部分由平台 bear

serviceFee = 按 rebate_rule.service_fee_value (默认 0)
            // 不显示给学生,内部记账用

netPayAmount = listAmount - couponAmount - pointsDeductAmount
             // 必须 >= 0,否则抛 AMOUNT_INVALID

if netPayAmount == 0:
    orderType = 'zero_price'
    跳过支付直接 completed
else:
    orderType = 'normal'
    走支付流程
```

**返点计算** (写到 order_item.rebate_share):
```
rebateBase = max(0, listAmount - serviceFee)
rebateAmount = rebateBase * rebate_rule.rebate_value  (rate 模式)
             或 rebate_rule.rebate_value              (fixed 模式)
```
