# WEB-ADMIN 导航与菜单优化反馈（给主Agent）

## 1. 任务背景与目标
- 当前管理端存在三类高频问题：
  - `竞赛通管理` 展开后菜单过长，阅读与点击效率低；
  - 混合导航下可能出现双菜单树，造成重复入口与认知干扰；
  - 首页/看板入口若与实际菜单注册不一致，易触发 404。
- 本轮目标是：在不改变权限模型的前提下，完成可用性修复与稳定性兜底，保证导航可达、可用、可维护。

## 2. 已完成优化清单（含验证）
- 顶部菜单防超长已加固：
  - 限高滚动（桌面与移动均受控）；
  - 桌面端长菜单双列显示；
  - 移动端单列触控可点；
  - 通用 fallback 已加，预防其他菜单未来变长。
- `/jst` 菜单自动分组父菜单已落地（前端路由组装层）：
  - 核心运营
  - 用户与渠道
  - 营销与权益
  - 积分与商城
  - 订单与财务
  - 风控与审计
- 既有前端去重与可达性兜底逻辑保持有效：
  - 重复业务根菜单去重；
  - 不可达入口禁用与提示，避免直接落入 404。
- 构建验证：
  - `npm run build:prod` 已通过；
  - 仅剩 bundle 体积 warning（非阻断，历史类问题）。

## 3. 关键结论：是否还需要前端控制码
- 权限真源应保持在后端：
  - `sys_menu`（菜单可见与路由来源）；
  - 后端接口鉴权（真正安全边界）。
- 前端控制仍然需要，但定位为 UX/容错层：
  - 路由可达判断；
  - 禁用态与友好提示；
  - 菜单去重；
  - 长菜单防溢出与分组展示。
- 明确原则：前端控制不承担安全职责，不替代后端鉴权。

## 4. 当前问题与风险
- 若线上仍显示旧样式，首因通常是静态资源缓存（浏览器/CDN）未刷新。
- 若新增菜单不在当前分组规则内，将进入“其他功能”分组（属预期兜底，但需关注信息架构）。
- 历史菜单命名或编码不规范时，规则命中可能受影响（尤其标题匹配类逻辑）。

## 5. 给主Agent的后续请求（资源清单）
- 决策项：
  - 是否将前端分组策略固化为数据库真实父菜单（SQL 层），实现长期一致治理。
- 资源项：
  - 提供可执行菜单迁移的测试库权限（`sys_menu/sys_role/sys_role_menu`）；
  - 提供回归账号：
    - `admin`（全量权限）
    - 受限运营账号（验证禁用态与提示）
- 发布后回归清单：
  - 顶部菜单展开与“更多菜单”行为；
  - 首页/看板入口跳转；
  - 无权限入口禁用与提示；
  - 移动端菜单可点击性与可滚动性。

## 6. 建议验收标准
- `竞赛通管理` 展开不超过视口，支持滚动。
- 不出现重复菜单树。
- 首页/看板入口无 404。
- 移动端菜单可操作（单列展示、触控可点）。

## 7. 说明（边界与口径）
- 本报告口径：权限仍由后端与数据库主导，前端负责展示稳定性与用户体验。
- 本次交付面向主Agent，后续如需并入 SQL 真分组方案，可在现有策略基础上平滑迁移。
