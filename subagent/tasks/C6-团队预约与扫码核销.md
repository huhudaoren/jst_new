## 业务背景

PRD 5.6/7.7/8.12 定义的"线下活动预约 + 现场多码核销"能力。
- **团队预约**：渠道方一次为 N 个学生（含未注册临时档案）批量预约某场次，系统生成团队单 + 逐个成员子订单（复用 C2 订单逻辑）。
- **核销子项**：每个成员/每个预约可挂 N 个独立子二维码（到场/礼品/仪式/自定义），**每个码独立状态流转互不影响**。
- **本任务范围**：团队预约创建 + 扫码核销（支持团队 & 个人）+ 状态机推进。**C7 只做个人预约的独立入口**（本任务底层已支持个人，C7 是包装 UI + 个人支付闭环）。

## 状态机

**SM-11 个人预约 `jst_appointment_record.main_status`**
```
unbooked → booked → partial_writeoff → fully_writeoff
                  ↘ expired / cancelled / partial_writeoff_ended
```

**SM-12 核销子项 `jst_appointment_writeoff_item.status`**
```
unused → used
       → expired
       → voided (admin 作废)
```

**SM-13 团队预约 `jst_team_appointment.status`**
```
booked → partial_writeoff → fully_writeoff
                          → partial_writeoff_ended (终态)
```

联动规则：
- 任一子项 `unused → used` 时：若该子项属于团队预约，需在 `lock:team_appt` 下累加 `writeoff_persons`（按成员维度而非子项维度，同一成员所有核销子项 used 后计 1 人次）
- 当 `writeoff_persons == total_persons` → team status = `fully_writeoff`
- 全部个人预约子项 used → individual status = `fully_writeoff`

## 锁（必须登记）

- `lock:team_appt:{teamAppointmentId}` — 团队维度计数互斥（等 3s，持 5s）
- `lock:writeoff:item:{itemId}` — 单码核销互斥（等 2s，持 3s）
- `lock:appointment:book:{contestId}:{sessionCode}` — 场次名额扣减（等 3s，持 5s）

## 接口契约

### 渠道方创建团队预约
**POST /jst/wx/team-appointment/apply** （role `jst_channel`）
- 入参：
  ```
  { contestId, sessionCode, appointmentDate,
    members: [{ userId?, participantId?, subOrderInput: {...} }],
    invoiceInfo? }
  ```
- 逻辑：@Transactional + 场次锁
  1. 查 contest 校验 `support_appointment=1` 且 `appointment_date` 合法
  2. 锁 `lock:appointment:book:{contestId}:{sessionCode}`
  3. 校验剩余名额（contest.appointment_capacity - 已 booked）
  4. 生成 `team_no`，INSERT `jst_team_appointment` status=`booked`，total_persons=members.size
  5. 对每个成员：
     - 调 `OrderService.createOrderForAppointment(member.subOrderInput)` 生成 `jst_order_main`（复用 C2 下单，禁用积分/券简化本期）
     - INSERT `jst_team_appointment_member` 关联 sub_order_id
     - INSERT `jst_appointment_record` (team_id, user_id/participant_id, main_status=`booked`, qr_code)
     - 按 contest.writeoff_config 为每条 appointment 生成 N 条 `jst_appointment_writeoff_item` (status=`unused`, qr_code=UUID)
  6. 返回 `TeamAppointmentApplyResVO { teamAppointmentId, teamNo, memberCount, subOrderIds[] }`

### 渠道方查团队预约列表 / 详情
- **GET /jst/wx/team-appointment/list**
- **GET /jst/wx/team-appointment/{id}** 含 members 及每人 writeoff 进度

### 学生查自己的预约
- **GET /jst/wx/appointment/my** — 个人预约列表（含 team 归属标记）
- **GET /jst/wx/appointment/{id}** — 含 writeoff_items 各子项二维码和状态

### 扫码核销（赛事方/admin）
**POST /jst/wx/writeoff/scan** （role `jst_partner` 或 `jst:event:writeoff:scan`）
- 入参：`{ qrCode, scanType? }`
- 逻辑：@Transactional
  1. 解析 qrCode（UUID 查 `jst_appointment_writeoff_item`）找不到 → `WRITEOFF_QR_INVALID`
  2. 校验 scanner 所属 partner_id == contest.partner_id（PartnerScope）
  3. `lock:writeoff:item:{itemId}`
  4. 校验 status=`unused`；非法 → `WRITEOFF_ITEM_STATUS_INVALID`
  5. `UPDATE item SET status='used', writeoff_time=NOW(), writeoff_user_id=?`
  6. 检查该成员所有子项是否全 `used`，若是：
     - 推进 `jst_appointment_record.main_status` 按 SM-11
     - 若该 record 属团队，则 `lock:team_appt:{teamId}` + `writeoff_persons++`，若达 total → team `fully_writeoff`
  7. 返回 `{ itemId, itemType, memberName, appointmentStatus, teamStatus? }`

### 核销记录列表
- **GET /jst/wx/writeoff/records** 分页，filter by partnerId/contestId/date
