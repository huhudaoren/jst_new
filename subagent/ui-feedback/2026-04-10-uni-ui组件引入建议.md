# 建议：引入 uni-ui 组件替代手写 CSS

> **提交者**：UI Polish Agent（专职视觉优化）
> **职责**：对照原型 PNG 逐页重构样式，不改业务逻辑/后端。
> **发现**：`uni_modules/` 已安装完整 uni-ui 组件库，但全部页面零使用，所有导航栏/搜索框/列表/标签/徽章/步骤条均手写 CSS。

## 现状问题
1. 每个页面手写 `__nav / __back / __nav-title / __nav-placeholder` 四件套导航栏，样式各异
2. 搜索框、Tab、标签、徽章在 40+ 页面各自实现，维护成本高
3. UI Polish Agent 按规则不能改 `<script>` 里的 import/components 注册，无法自行引入

## 建议主 Agent 安排
按优先级逐步替换，每次一个组件类型：

| 优先级 | uni-ui 组件 | 替换目标 | 影响页面数 |
|---|---|---|---|
| 高 | `uni-nav-bar` | 手写 `__nav / __back / __title` 导航栏 | ~30 页 |
| 高 | `uni-search-bar` | 手写搜索框 | ~8 页 |
| 高 | `uni-icons` | emoji 图标（🔍📝📦 等）→ 统一图标库 | 全部 |
| 中 | `uni-tag` | 手写 `.chip / .tag / .badge` 标签 | ~20 页 |
| 中 | `uni-badge` | 手写红点徽章 | ~5 页 |
| 中 | `uni-list` + `uni-list-item` | my/index 的 cell 列表 | ~10 页 |
| 中 | `uni-steps` | order-detail 的时间线 | 2 页 |
| 低 | `uni-section` | 手写 section-title | ~15 页 |
| 低 | `uni-segmented-control` | 手写 switcher/segment | ~5 页 |

## 收益
- 视觉一致性自动保证（uni-ui 内部样式统一）
- 代码量减少 40%+（删除大量手写导航/搜索/标签 CSS）
- 暗色模式/主题切换免费获得（uni-ui 支持 CSS 变量）
- 无障碍支持（uni-ui 组件内置 aria 属性）

## 执行方式
建议由后端/前端 Agent 分批执行（需改 script import），UI Polish Agent 后续对齐样式覆写（`/deep/` 或 `::v-deep` 微调）。

## 优先级
中（不阻塞当前视觉对齐，但长期维护收益大）
