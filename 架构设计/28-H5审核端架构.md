# 28. H5 审核端架构（决策建议）

> PRD V4.1 列出 5 个终端形态，其中"平台 H5 审核端"在 V3.0 原型已设计 11 个页面。
> 本文档**不直接给实施方案**，而是给出 3 种选型分析，由你拍板。

---

## 1. 背景

V3.0 原型 `小程序原型图/admin-*.html` 共 11 个页面：
- admin-login.html / admin.html
- admin-identity.html / admin-identity-detail.html (身份认证审核)
- admin-enroll.html / admin-enroll-detail.html (报名材料审核)
- admin-voucher.html (对公转账凭证审核)
- admin-partner-review.html (赛事方入驻审核)
- admin-users.html (用户列表)
- admin-score-publish.html (成绩发布)
- admin-notice-edit.html (公告编辑)
- admin-maic.html (AI 课程审核)
- admin-channel-settlement.html / admin-event-settlement.html (结算审核)

**用途**：让运营/审核人员**在手机上随时审核**，不必打开 PC RuoYi。

---

## 2. 三种选型对比

### 方案 A：嵌入小程序作为「审核员」分包 ⭐ 推荐

**做法**：在 jst-uniapp 项目内新增 `pages-sub/admin/` 子分包，通过角色 `jst_platform_op` 控制访问。

| 优 | 缺 |
|---|---|
| 工程成本最低（共用 Uniapp） | 包体增大 ~500KB |
| 复用同一登录 + token + 网络层 | 需要在小程序权限菜单暴露管理员入口 |
| 复用 jst-common UI 组件 | 微信合规：管理类功能不能太显眼 |
| 一套打包发布 | |

**接口前缀**：`/jst/wx/admin/**`，权限 `hasRole('jst_platform_op')`
**入口**：登录页底部「管理员登录」按钮（与 V3.0 原型 admin-login.html 一致）

### 方案 B：独立 Vue3 H5 工程

**做法**：新建 `jst-h5-admin/`，纯 H5 单页应用，独立部署到 admin.jst.example.com。

| 优 | 缺 |
|---|---|
| 与小程序解耦，发版独立 | 工程数 +1，维护成本高 |
| 不受微信审核限制 | 无微信支付/wx.login |
| 可用 Vue3 + Vite 现代栈 | 登录需另写（账号密码） |

### 方案 C：本期不做，全用 PC RuoYi-Vue 端审核

**做法**：把 11 个 H5 页面对应功能在 RuoYi-Vue admin 端用 element-ui 实现（PC 端已有）。审核员用电脑做事。

| 优 | 缺 |
|---|---|
| 工程成本最低（已有 PC 端） | 用户体验差（手机审核场景缺失） |
| 单一认证体系 | 合作运营 OKR 可能要求移动审核 |
| 当前生成的 50 张 CRUD 已覆盖 80% | |

---

## 3. 我的推荐

**MVP 上线前：方案 C**
- 当前 50 张表的 PC RuoYi 端 CRUD 已完成，运营完全可在 PC 操作
- 节省 1-2 周工程成本，把资源集中在资金主线
- H5 审核留 P2 待后续

**MVP 后扩展期：方案 A**
- 在小程序内加 admin 分包
- 复用账号体系
- 11 个页面按原型实现
- 工程改动小

**不推荐方案 B**：维护成本不划算

新增 store：`store/admin.js` 切换审核员模式。
