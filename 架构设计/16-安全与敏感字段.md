# 16. 安全设计与敏感字段处理

> 范围：登录认证、敏感字段加密、数据权限、越权防御、接口安全
> 强制：所有接口在 Code Review 阶段必须对照本文档自检

---

## 1. 认证体系

### 1.1 管理后台 Web（PC）
沿用若依原生：账号密码 + 验证码 + JWT。无需改造。

### 1.2 微信小程序
新增 `WxAuthService`，落点 `jst-user/com.ruoyi.jst.user.auth`：
```
POST /jst/wx/auth/login
  入参：code (wx.login 拿到)
  逻辑：code → openid/unionid → 查/建 jst_user → 颁发若依 JWT
  返回：token + userInfo + isNewUser(若新用户引导完善资料)
```
**关键约束**
- 小程序 token 与若依 PC token 共用一套（`Authorization: Bearer xxx`）
- token 续期沿用若依机制
- **首次登录不得自动写入 mobile**，须用户主动绑定（getPhoneNumber 后用户授权）

### 1.3 赛事方 Web 工作台
独立登录页 `/organizer/login`，复用若依用户体系，但通过 **角色** 隔离：`role_key='ROLE_PARTNER'`。

### 1.4 渠道方
微信小程序内同账号体系，通过 `user_type='channel'` 区分。

## 2. 敏感字段加密（AES-256）

### 2.1 加密字段清单（DB 中均为 VARCHAR(128) 密文）

| 表 | 字段 | 加密类型 |
|---|---|---|
| jst_user | id_card_no | AES |
| jst_participant | id_card_no | AES |
| jst_channel | id_card_no, bank_account_no | AES |
| jst_event_partner | settlement_info_json 内 bank_account_no 字段 | 字段级 AES |
| jst_payment_record | voucher_url | 不加密（OSS URL，但 bucket 私有） |

### 2.2 实现

```java
package com.ruoyi.jst.common.crypto;

@Component
public class JstCipher {
    @Value("${jst.security.aes-key}") private String aesKey;
    public String encrypt(String plain) { /* AES/CBC/PKCS5Padding + IV 拼接 base64 */ }
    public String decrypt(String cipher) { /* 反向 */ }
}
```

**约束**
- 密钥从环境变量 `JST_AES_KEY` 注入，**禁止入仓**

#### AES Key 运维生成步骤（部署前一次性）

```bash
# 1. 在运维主机上生成 32 字节强随机 key（256-bit AES）
openssl rand -base64 32
# 输出示例：xK9aB7zP3vQ1mR5sT8wY2cN4dF6hJ0lM... (44 字符 base64)

# 2. 写入运维密钥管理（任选其一）：
#    - 阿里云 KMS / Hashicorp Vault（推荐）
#    - 服务器 systemd EnvironmentFile：/etc/jst/jst.env
#      JST_AES_KEY=xK9aB7zP3vQ1mR5sT8wY2cN4dF6hJ0lM...
#    - Docker secret / K8s Secret

# 3. 启动应用时通过环境变量加载（systemd 示例）
[Service]
EnvironmentFile=/etc/jst/jst.env
ExecStart=/usr/bin/java -jar /opt/jst/ruoyi-admin.jar

# 4. 验证：
#    应用启动后写入一条带敏感字段的测试数据，再 SELECT 应得到密文
#    再走 Service 层读取应得到明文

# 5. 密钥轮换（每年一次）：
#    a. 准备新 key JST_AES_KEY_NEW
#    b. 上线"双密钥解密"版本：先用旧 key 解，失败用新 key 解；写入用新 key
#    c. 跑批：全表 SELECT 明文 → 用新 key 重新加密 UPDATE
#    d. 撤销旧 key
#
# 重要禁止事项：
# - 禁止将 key 写入 application.yml / application-prod.yml
# - 禁止将 key 提交 git（包括 .env、.env.example）
# - 禁止在日志中打印 key 或包含 key 的环境变量 dump
# - 禁止将 key 通过 IM / 邮件 明文传递，须用密码管理工具（1Password/Bitwarden）
```
- 加密字段在 Mapper 层通过 `@Result(typeHandler=AesTypeHandler.class)` 自动加解密，业务层用明文
- **日志中禁止打印解密后的明文**（用 `MaskUtil.idCard()` / `MaskUtil.mobile()` 脱敏）
- **接口出参禁止返回明文**：列表场景显示掩码 `1101**********1234`，详情场景按权限位 `jst:user:viewSensitive` 控制

### 2.3 脱敏工具

```java
package com.ruoyi.jst.common.util;

public class MaskUtil {
    public static String mobile(String m) {
        return m == null || m.length() < 7 ? m : m.substring(0,3) + "****" + m.substring(7);
    }
    public static String idCard(String id) {
        return id == null || id.length() < 10 ? id : id.substring(0,4) + "**********" + id.substring(id.length()-4);
    }
    public static String bankAccount(String acc) {
        return acc == null || acc.length() < 8 ? acc : acc.substring(0,4) + "********" + acc.substring(acc.length()-4);
    }
}
```

## 3. 数据权限

### 3.1 平台管理员
全量数据，无过滤。沿用若依 `@DataScope` 注解。

### 3.2 赛事方
**只能看到自己的数据**：
- `jst_contest WHERE partner_id = currentUser.partnerId`
- `jst_enroll_record WHERE contest_id IN (我的赛事)`
- `jst_event_settlement WHERE partner_id = currentUser.partnerId`

**实现方式**：在每个 `*Mapper.xml` 的查询 SQL 中**强制**追加 partner_id 过滤；通过 `@PartnerScope` 注解 + AOP 自动注入，禁止 Service 层手写过滤。

### 3.3 渠道方
**只能看到自己拉新/发起的数据**：
- `jst_user` 通过 `bound_channel_id = currentUser.channelId` 过滤
- `jst_order_main WHERE channel_id = currentUser.channelId`
- `jst_rebate_ledger WHERE channel_id = currentUser.channelId`
- `jst_rebate_settlement WHERE channel_id = currentUser.channelId`

通过 `@ChannelScope` 注解 + AOP 实现。

### 3.4 学生/家长
**只能看到自己的数据**：所有查询接口必须按 `user_id = currentUser.userId` 过滤。**禁止接受前端传入的 userId 参数**。

## 4. 越权防御

### 4.1 横向越权（IDOR）防御
**强制规范**：所有「按 ID 取详情/编辑/删除」接口在 Service 层 **第一行** 必须做归属校验：

```java
public OrderDetailVO getDetail(Long orderId) {
    OrderMain order = orderMainMapper.selectById(orderId);
    if (order == null) throw new ServiceException("订单不存在");
    // 关键：归属校验
    SecurityCheck.assertSameUser(order.getUserId());
    return convert(order);
}
```

`SecurityCheck.assertSameUser(targetUserId)` 工具：
```java
public static void assertSameUser(Long targetUserId) {
    Long currentId = SecurityUtils.getUserId();
    if (!Objects.equals(currentId, targetUserId)) {
        throw new ServiceException("无权访问该资源", 403);
    }
}
```

### 4.2 纵向越权（权限位）
所有 Controller 方法必须显式 `@PreAuthorize("@ss.hasPermi('jst:xxx:yyy')")`，**禁止无权限标识的业务接口**。

### 4.3 表单越权
学生提交动态表单时，禁止前端传 `participant_id`，必须由后端从当前 user 已认领的 participant 列表中选择。

## 5. 接口安全

### 5.1 防重放
所有支付/退款/打款类敏感接口必须带 `nonce` + 时间戳，AOP 校验 5 分钟内 nonce 不重复（Redis key `jst:nonce:{userId}:{nonce}` TTL 5min）。

### 5.2 参数校验（强制）
所有 ReqDTO 必须：
- 标 `@NotNull` / `@NotBlank` / `@Size` / `@Min` / `@Pattern`
- Controller 方法上 `@Validated`
- 全局 `MethodArgumentNotValidException` handler 返回 400 + 字段级错误

### 5.3 SQL 注入
- 全部走 MyBatis `#{}` 参数化，**禁用 `${}`**（除排序字段外）
- 排序字段必须白名单校验：
```java
private static final Set<String> ALLOWED_SORT = Set.of("create_time","pay_time","amount");
```

### 5.4 XSS
- 富文本字段（公告 content / 赛事 description）入库前用 `Jsoup.clean()` 白名单过滤
- 所有用户输入字符串字段在 `BaseEntity` 反序列化时统一 `EscapeUtil.clean()`

### 5.5 文件上传
- 走 jst-common 的 `OssService.uploadWithCheck()`，做：MIME 校验 + 大小限制 + 病毒扫描钩子（暂不接 ClamAV，留 SPI）
- 上传 bucket **必须私有**，访问通过 **签名 URL** + 短 TTL（图片 1h，凭证/合同 10min）
- 禁止前端直传

## 6. 审计日志（强制）

所有「写操作」必须落 `jst_audit_log`，通过 `@OperateLog` 注解 + AOP 实现：
```java
@OperateLog(module="订单", action="REFUND_APPROVE", target="#{refundId}")
@PreAuthorize("@ss.hasPermi('jst:order:refund')")
public AjaxResult approveRefund(@RequestBody RefundApproveReqDTO req) { ... }
```

**before/after JSON 必须脱敏**（调用 MaskUtil）后再写 audit_log。

## 7. 风控触点

| 触点 | 检查项 | 命中动作 |
|---|---|---|
| 用户注册 | 同一手机/设备号 24h 内注册次数 > 5 | warn |
| 渠道绑定切换 | 同一 user 7 天内切换次数 > 3 | manual 审核 |
| 退款申请 | 同一 user 30 天内退款 > 5 笔 | manual 审核 |
| 优惠券领取 | 同一 user 单日领取 > 10 张 | intercept |
| 零元订单 | 同一 user 单日零元单 > 20 | intercept |

落点：`jst-risk` 的 `RiskCheckService`，在关键 Service 入口处显式调用。
