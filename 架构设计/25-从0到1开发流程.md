## 1. 全局开发顺序（推荐 Agent 流水线）

```
阶段 A：基础设施 (单个 Agent 串行)
  A1. jst-common 通用基建
       └─ 业务异常码 BizErrorCode (枚举)
       └─ 雪花 ID 生成器 SnowflakeIdWorker
       └─ Redisson 配置 RedissonConfig + JstLockTemplate
       └─ AES 加解密 JstCipher + AesTypeHandler
       └─ MaskUtil 脱敏工具
       └─ 阿里云 OSS 工具 OssService
       └─ 状态机抽象 StateMachine<E extends Enum>
       └─ @OperateLog 注解 + AOP（写 jst_audit_log）
       └─ @PartnerScope / @ChannelScope 数据权限注解 + AOP

阶段 B：领域基线 (各模块串行，模块内可并行)
  B1. jst-user：注册/登录/资料/临时档案/认领映射 (SM-14, SM-15)
  B2. jst-event：赛事方档案/赛事 CRUD/动态表单模板 (SM-4, SM-5, SM-25)
  B3. jst-points：积分账户/流水/等级/规则 (基础查询，无消耗)
  B4. jst-marketing：优惠券/权益基础 CRUD (无核销)
  B5. jst-organizer：入驻申请审核 (SM-3, SM-4)

阶段 C：交易主线 (核心，必须严格串行)
  C1. jst-event：报名记录(form_snapshot_json) (SM-6)
  C2. jst-order：订单创建(混合支付分摊) + Mock 微信支付 (SM-1 created→paid)
  C3. jst-points：积分账户扣减/冻结 + 乐观锁 (与 C2 联动)
  C4. jst-marketing：优惠券锁定/使用/回退 (与 C2 联动)
  C5. jst-order：退款全流程 (SM-2 + SM-1 资金回退)
  C6. jst-channel：返点规则 + 计提台账 (SM-8) + 渠道提现负向抵扣 (SM-9)
  C7. jst-event：成绩录入/审核/发布 (SM-19) + 证书发放 (SM-20)
  C8. jst-finance：赛事方结算 (SM-10) + 合同/发票/打款

阶段 D：预约与核销 (高并发，独立交付)
  D1. jst-order：个人预约 (SM-11)
  D2. jst-order：团队预约创建 + 成员明细
  D3. jst-order：扫码核销（团队部分核销 SM-13 + 子项 SM-12）⭐
  D4. Quartz Job：J03 场次结束扫描

阶段 E：增值与生态
  E1. jst-points：积分商城 (SM-18)
  E2. jst-event：课程 + AI 课程 + 阿里云 VOD
  E3. jst-message：消息模板/发送/重试 (J09)
  E4. jst-risk：风控规则/预警/工单 (SM-24)

阶段 F：定时任务接入
  F1~F12：J01~J12 全部 Quartz Job 入库

阶段 G：前端
  G1. RuoYi-Vue 管理后台 Vue 页面（用代码生成器输出）
  G2. Uniapp 小程序（学生/家长/渠道）
  G3. Uniapp 工作台（赛事方）
```

**单 Agent 模式**：按 A→B→C→D→E→F→G 顺序，每完成一个编号项立刻跑测试 + commit
**多 Agent 模式**：阶段 A 必须独立完成；阶段 B 内部 5 个 Agent 可并行；阶段 C 必须串行（资金主线）；阶段 D/E 可并行

### 1.1 ⚠️ 关于"单 Agent 0→1 完成全部"的现实评估

**结论：单 Agent 无法独立完成全部 51 表 / 11 模块 / 三端**。强烈建议「主 Agent 协调 + 子 Agent 派发 feature + 人工 5 个 Gate」混合模式，原因：

| 限制 | 影响 |
|---|---|
| 上下文窗口物理上限 | 即使 1M context，跑到第 8-15 个 feature 必然出现"早期记忆衰减"，开始忘记状态机枚举名/锁名/命名规范 |
| 跨阶段强依赖 + 不可逆 | C2 依赖 C1 已 Mock 跑通；C3 依赖 C2 事务边界。单 Agent 没有"回头检查"机制 |
| 代码生成器+人工剪切环节 | 决策 Q6=A 要求人工去管理后台跑 gen → 剪切。Agent 无法完成 |
| 联调依赖外部资源 | DB 导入、阿里云 AK、微信支付切换、真机调试，都必须人工 |

#### 推荐执行模式

```
主 Agent（持有全部架构契约 11-25 文档）
   │
   ├─ 派发 子Agent A1（jst-common 基建）→ 主审查 → Gate1 人工编译
   ├─ 派发 子Agent B1（jst-user 注册+认领）→ 主审查 → Gate2 人工 gen+联调
   ├─ 派发 子Agent B2/B3/B4 并行（赛事/积分/营销基线）
   ├─ 派发 子Agent C1-C5 严格串行（资金主线）→ Gate4 资金联调（最高风险）
   ├─ 派发 子Agent D/E 并行（核销/商城/课程/消息/风控）
   └─ 派发 子Agent F（Quartz 入库）→ Gate5 三端联调
```

**5 个人工 Gate**（不可省略）
- **Gate 1**：`mvn clean install` 通过，模块 pom 全部可解析
- **Gate 2**：jst-common 基建单元测试通过
- **Gate 3**：第一个完整 feature 联调通（建议 jst-user 注册→临时档案认领，覆盖 SM-14 + 防孤儿规则）
- **Gate 4**：资金主线 C1-C5 联调通（订单→支付→退款→返点），**这是项目最大风险点**
- **Gate 5**：三端联调通（管理后台 + 小程序 + 赛事方工作台）

**子 Agent 派发原则**
1. 每次派发只交付 1 个 feature（≤ 5 个 Java 文件 + 1 个 Vue 文件）
2. 必须先读 13/15/16/25 + 当前 feature 涉及的状态机/API 文档
3. 必须输出"完成清单"含：新增文件路径、关联 SM、关联表、关联锁名、自检结果
4. 主 Agent 维护一份"已完成 feature 注册表"，避免子 Agent 重复造轮子或命名冲突

### Step 4. 实现状态机
```java
package com.ruoyi.jst.user.enums;

public enum ClaimStatus {
    UNCLAIMED, AUTO_CLAIMED, MANUAL_CLAIMED, PENDING_MANUAL;

    private static final Map<ClaimStatus, Set<ClaimStatus>> ALLOWED = new HashMap<>();
    static {
        ALLOWED.put(UNCLAIMED,      Set.of(AUTO_CLAIMED, MANUAL_CLAIMED, PENDING_MANUAL));
        ALLOWED.put(PENDING_MANUAL, Set.of(MANUAL_CLAIMED, UNCLAIMED));
        ALLOWED.put(AUTO_CLAIMED,   Set.of(UNCLAIMED));
        ALLOWED.put(MANUAL_CLAIMED, Set.of(UNCLAIMED));
    }

    public void assertCanTransitTo(ClaimStatus target) {
        if (!ALLOWED.getOrDefault(this, Set.of()).contains(target)) {
            throw new ServiceException(
                String.format("非法档案认领状态跃迁: %s → %s", this, target),
                BizErrorCode.JST_USER_CLAIM_ILLEGAL_TRANSIT.code());
        }
    }
}
```

### Step 5. 实现 Service（必须含锁 + 事务 + 审计）
```java
@Service
public class ParticipantClaimServiceImpl implements ParticipantClaimService {

    @Autowired private ParticipantMapper participantMapper;
    @Autowired private ParticipantUserMapMapper mapMapper;
    @Autowired private JstLockTemplate jstLockTemplate;

    @Override
    @Transactional(rollbackFor = Exception.class)
    @OperateLog(module = "用户", action = "CLAIM_AUTO", target = "#{participantId}")
    public void claimByAuto(Long participantId, Long userId) {
        jstLockTemplate.execute("lock:claim:participant:" + participantId, 3, 5, () -> {
            Participant p = participantMapper.selectById(participantId);
            if (p == null) throw new ServiceException("档案不存在");

            // 状态机校验
            ClaimStatus.valueOf(p.getClaimStatus().toUpperCase())
                .assertCanTransitTo(ClaimStatus.AUTO_CLAIMED);

            // 写 map 表
            ParticipantUserMap map = new ParticipantUserMap();
            map.setParticipantId(participantId);
            map.setUserId(userId);
            map.setClaimMethod("auto_phone_name");
            map.setClaimTime(DateUtils.getNowDate());
            map.setStatus("active");
            mapMapper.insert(map);

            // 更新 participant
            p.setClaimStatus("auto_claimed");
            p.setClaimedUserId(userId);
            p.setClaimedTime(DateUtils.getNowDate());
            participantMapper.updateById(p);

            return null;
        });
    }
}
```

### Step 6. 实现 Controller（带权限 + 校验 + 数据权限）
```java
@RestController
@RequestMapping("/jst/user/participant")
public class ParticipantController extends BaseController {

    @Autowired private ParticipantService participantService;
    @Autowired private ParticipantClaimService claimService;

    /**
     * 临时档案-列表查询
     * 权限：jst:user:participant:list
     * 数据权限：渠道方仅可见自己创建的档案（@ChannelScope 自动过滤）
     */
    @PreAuthorize("@ss.hasPermi('jst:user:participant:list')")
    @ChannelScope(field = "createdByChannelId")
    @GetMapping("/list")
    public TableDataInfo list(@Validated ParticipantQueryReqDTO query) {
        startPage();
        List<ParticipantListResVO> list = participantService.selectList(query);
        return getDataTable(list);
    }

    /**
     * 临时档案-管理员手动认领
     * 权限：jst:user:participant:claim
     */
    @PreAuthorize("@ss.hasPermi('jst:user:participant:claim')")
    @PostMapping("/claim/admin")
    public AjaxResult claimByAdmin(@Validated @RequestBody ClaimAdminReqDTO req) {
        claimService.claimByAdmin(req.getParticipantId(), req.getUserId(), req.getReason());
        return AjaxResult.success();
    }
}
```

## 3. 强约束清单（违反 = 退回重做）

### 3.1 命名
- 包路径：`com.ruoyi.jst.{module}.{layer}`，**禁止** `com.ruoyi.jst.{layer}.{module}`
- 类名：`{Entity}Controller / {Entity}Service / {Entity}ServiceImpl / {Entity}Mapper`
- DTO/VO 后缀：`ReqDTO`、`Form`、`ResVO`，**禁止** `XxxBO`、`XxxParam`、`XxxResult`
- Mapper.xml namespace：与 Mapper 接口完全一致

### 3.2 注释
- **每个**类必须有 javadoc：包含中文功能说明 + `@author` + `@since`
- **每个**业务方法必须有 javadoc：含 `@param` / `@return` / 关联表 / 关联状态机 / 关联权限
- 复杂逻辑（>20 行）必须有行内中文注释
- 状态机跃迁、加锁位置、事务边界，**必须** 有 `// SM-x`、`// LOCK: ...`、`// TX:` 标注

### 3.3 异常
- 业务错误：抛 `ServiceException("中文消息", BizErrorCode.XXX.code())`
- 系统错误：抛 `RuntimeException`（由全局 handler 兜底）
- **禁止** 吃异常（catch 不处理）
- **禁止** `e.printStackTrace()`，必须 `log.error("...", e)`

### 3.4 数据访问
- **禁止** 在 Service 直接拼 SQL
- **禁止** 在 Controller 调 Mapper（必须经 Service）
- **禁止** 跨模块直接 import 别的模块的 Mapper（必须通过该模块的 Service）

### 3.5 事务
- 资金/状态机方法必须 `@Transactional(rollbackFor = Exception.class)`
- 事务方法内**禁止**调用 RPC、发短信、推消息（用 `TransactionSynchronizationManager.registerSynchronization` 在 commit 后异步触发）

### 3.6 锁
- 必须用 `JstLockTemplate.execute()`，**禁止** 裸用 RedissonClient
- 锁名必须在 `15-Redis-Key与锁规约.md` 登记，未登记的需先 PR 修文档

### 3.7 日志
- 类内统一 `private static final Logger log = LoggerFactory.getLogger(XxxClass.class);`
- 每个 Job 入口必须打印 `total/success/fail/cost`
- **禁止** 打印明文 mobile/idcard/bank（用 MaskUtil）

### 3.8 测试数据规约 ⭐ 强制
**禁止任何形式的页面内 Mock / 假数据 / 硬编码列表**。具体要求：

1. **前端**（Vue 管理后台 / Uniapp 小程序）
   - ❌ 禁止在 `data()` / `state` / `computed` 中写死示例数据
   - ❌ 禁止 `if (!列表数据.length) 列表数据 = [{id:1, name:'测试'}]`
   - ❌ 禁止注释掉接口调用、改用本地 mock json
   - ✅ 即使后端接口未就绪，也必须 **在数据库中先 INSERT 真实测试记录**，前端通过真实接口读取
   - ✅ 接口未实现时，使用 13-技术栈文档定义的 Mock 实现类（如 `MockWxPayServiceImpl`）从 DB 读数据返回

2. **后端**
   - ❌ 禁止 Service 内 `return Arrays.asList(new XxxVO(...))` 之类硬编码
   - ❌ 禁止单元测试用 `Mockito.when(mapper.selectXxx).thenReturn(假对象)` 来"绕过"DB
   - ✅ 单元测试统一用 `@SpringBootTest + @Sql` 在 H2 或 MySQL test schema 中先插入测试数据，再走真实 Service 链路
   - ✅ 集成测试必须用真实 MySQL（profile=test）+ 真实 Redis

3. **测试数据落点**
   - 所有功能测试数据集中在 `D:\coding\jst_v1\架构设计\ddl\99-test-fixtures.sql`（按模块分段，**仅 dev/test 环境执行，禁止生产**）
   - 该文件由开发新增功能时累加，commit 时一并入仓
   - 命名约定：测试用户 mobile=`13800000xxx`，姓名前缀=`测试_`，便于清理

4. **为什么这条强制**
   - 防止前端"看上去能跑"实际接口未通的情况，避免上线前才发现联调问题
   - 防止脏数据混入生产
   - 强迫开发尽早暴露后端问题
   - 让 PM/测试可在管理后台直接 CRUD 测试数据，而不是去改前端代码

**违反检查方式**：CodeReview 阶段 grep
```
grep -r "mock\|MOCK\|测试数据\|hardcode\|硬编码\|return Arrays.asList" --include=*.{vue,js,ts,java}
```
有命中且非注释中的解释 → 必须删除并改为 DB 读取。

## 4. 与代码生成器的协作（决策 Q6=A）

### 4.1 流程
```
1. 在管理后台「代码生成」导入 jst_xxx 表
2. 修改生成信息：
   - 包路径 com.ruoyi.jst.{module}
   - 模块名 {module}（如 user/event/order）
   - 业务名 {entity}（如 participant/contest/order）
   - 功能名 中文菜单名
3. 下载 zip
4. 解压到临时目录
5. 手动拷贝（开发完成）：
   - main/java/com/ruoyi/{module}/* → jst-{module}/src/main/java/com/ruoyi/jst/{module}/*
   - main/resources/mapper/{module}/* → jst-{module}/src/main/resources/mapper/{module}/*
   - vue/index.vue → ruoyi-ui/src/views/jst/{module}/{entity}/index.vue
   - vue/api/{entity}.js → ruoyi-ui/src/api/jst/{module}/{entity}.js
   - menu.sql → 在管理后台执行
6. 按本文档 Step 3-9 改造代码
```

### 4.2 必须改包路径
若依生成器默认输出 `com.ruoyi.{module}`（**不带 jst 前缀**），开发拷贝时必须：
- IDE 全局替换：`com.ruoyi.{module}` → `com.ruoyi.jst.{module}`
- Mapper.xml 的 `namespace` 同步替换

### 4.3 不要直接编译生成的原始包
- 生成的代码**不要**直接放 ruoyi-admin 编译，必须先剪切到 jst-{module}

## 7. 紧急回滚策略

任何阶段出现以下情况立即停下：
- 状态机出现非法跃迁（数据库出现非法状态值）
- 资金账目对不上（订单实付 ≠ 支付记录之和）
- 临时档案被错误覆盖（user_id 被错误回写）
- Redisson 锁未释放导致大批量请求失败

回滚流程：
1. 停止写入（应用层开关 / 限流）
2. 备份当前 DB 快照
3. 用 audit_log 反向还原数据
4. 复盘根因，修补状态机/锁实现
5. 上线前必须新增对应回归测试
