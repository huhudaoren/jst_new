## 3. 建设目标

### 3.1 业务目标
- 建立统一的后台运营与财务管理中台，支撑竞赛通 V4.0 全部业务闭环。
- 承接渠道方、积分商城、赛事方入驻三大新增能力的复杂配置与日常运营。
- 通过独立后台权限体系，支持多角色、多菜单、多按钮、多数据范围的精细化权限控制。
- 提供可审计、可追溯、可导出的订单、积分、返点、结算与售后数据能力。

### 3.2 产品目标
- 平台管理员可在 PC 后台完成全部核心配置与管理。
- 运营可完成赛事、公告、积分、商品、消息模板等配置。
- 财务可完成渠道提现、赛事结算、发票、打款、对账处理。
- 审核人员可完成渠道认证、赛事方入驻、赛事提审、成绩证书等审核操作。

## 4. 技术与实现前提

### 4.1 技术基线
- 后台采用前后端分离架构实现。
- 管理 Web 前端使用独立后台前端工程开发。
- 后端接口由后台管理服务和业务服务共同提供。
- 菜单、路由、按钮权限通过统一权限配置中心动态下发。
- 管理端登录、权限、日志、导出、字典、参数配置采用独立后台基础能力实现。

### 4.2 落地约束
- 菜单路由遵循动态路由与权限控制机制。
- 页面按钮权限统一采用 `jst:模块:动作` 命名。
- 所有关键操作需写入后台操作审计日志。
- 所有金额类页面需支持导出 Excel。
- 所有规则配置页面需支持启停、版本记录或生效时间控制。
- 复杂配置放 PC 后台，不放 H5 审核端。

### 4.3 目录约定
- 前端页面建议按业务域拆分至 `admin-web/src/views/业务域/...`
- 前端 API 建议按业务域拆分至 `admin-web/src/api/业务域/...`
- 后端业务建议按领域独立拆分模块，避免将全部能力堆叠在单一基础模块中。

## 5. 管理角色与权限模型

### 5.1 角色定义

| 角色 | 说明 |
|------|------|
| 超级管理员 | 全量配置、审核、打款、系统维护 |
| 运营管理员 | 赛事、公告、消息、积分活动、商品运营 |
| 审核员 | 渠道认证、赛事方入驻、赛事审核、成绩证书审核 |
| 财务员 | 渠道提现、赛事结算、发票、打款、对账 |
| 积分运营 | 积分规则、成长值规则、等级权益、积分调整 |
| 商城运营 | 商品、库存、物流、售后 |
| 客服专员 | 用户查询、订单查询、售后跟进、消息处理 |

### 5.2 数据权限定义

| 角色 | 数据范围 |
|------|-------------|
| 超级管理员 | 全部数据 |
| 运营管理员 | 赛事、内容、商品、积分配置数据 |
| 审核员 | 审核任务相关数据 |
| 财务员 | 订单金额、提现、结算、发票相关数据 |
| 客服专员 | 用户、订单、售后、消息相关数据，只读为主 |

### 5.3 权限模型
- 菜单权限：控制页面是否可见。
- 按钮权限：控制新增、编辑、审核、导出、打款、调整等动作。
- 数据权限：控制可查看的数据范围。
- 典型按钮权限命名示例：
  - `jst:user:list`
  - `jst:user:edit`
  - `jst:channel:review`
  - `jst:rebateRule:add`
  - `jst:rebateWithdraw:review`
  - `jst:pointsRule:edit`
  - `jst:goods:updown`
  - `jst:eventPartner:review`
  - `jst:eventSettlement:pay`

## 8. 功能需求详述

### 8.1 管理驾驶舱

#### 页面目标
- 作为后台首页，展示全局经营指标、待处理任务、风险预警与快捷入口。

#### 核心内容
- KPI：今日报名、今日支付金额、待审核报名、待审核赛事、可提现金额、待结算金额。
- 趋势图：近 7 / 30 天报名趋势、支付趋势、积分消耗趋势。
- 待办卡片：渠道认证、赛事方入驻、赛事审核、成绩审核、证书审核、提现审核、售后待处理。
- 风险预警：库存不足、待审核积压、退款异常、负向返点台账。

#### 权限标识
- `jst:dashboard:view`

### 8.2 用户中心

#### 用户管理
- 支持按手机号、昵称、用户类型、注册时间、状态筛选。
- 列表字段：用户 ID、昵称、手机号、角色、当前等级、积分、成长值、绑定渠道方、注册时间、状态。
- 操作：查看详情、封禁 / 解封、调整积分、调整成长值、查看订单、查看售后。
- 权限标识：`jst:user:list`, `jst:user:edit`, `jst:user:adjust`, `jst:user:export`

#### 用户详情
- 基础资料、报名记录、课程记录、积分流水、成长值流水、兑换记录、消息记录。
- 客服备注、风险备注、黑名单标记。

#### 绑定关系管理
- 支持按学生、渠道方、绑定时间、当前状态筛选。
- 列表字段：学生姓名、手机号、渠道方名称、渠道方类型、绑定开始时间、失效时间、当前状态。
- 操作：查看变更历史、人工解绑、修正异常绑定。
- 关键规则提示：历史订单归属不因后续变更而回溯。
- 权限标识：`jst:binding:list`, `jst:binding:edit`, `jst:binding:export`

#### 用户消息记录
- 展示站内消息、短信、模板消息、触发来源、发送结果。
- 支持按消息类型、发送状态、用户范围检索。

### 8.3 渠道中心

#### 渠道认证审核
- 审核类型：老师 / 机构 / 个人。
- 列表字段：申请编号、申请人名称、类型、提交时间、审核状态、审核员。
- 详情页展示认证材料、申请说明、历史审核记录。
- 操作：通过、驳回、暂缓。
- 驳回必须填写原因。
- 权限标识：`jst:channelReview:list`, `jst:channelReview:review`

#### 渠道档案
- 展示渠道方基础信息、认证状态、绑定学生数、累计订单数、累计返点、当前等级、权益状态。
- 操作：启用 / 停用、查看绑定学生、查看订单、查看台账。
- 支持维护备注、渠道标签、风控状态。
- 权限标识：`jst:channel:list`, `jst:channel:edit`, `jst:channel:export`

#### 返点规则管理
- 配置颗粒度：赛事 + 渠道方。
- 支持返点模式：比例返点、固定金额返点。
- 支持平台服务费模式：固定金额、比例、不收取。
- 支持配置生效时间、失效时间、启用状态。
- 列表字段：赛事名称、渠道方名称、返点模式、返点值、服务费规则、生效时间、状态。
- 操作：新增、编辑、复制、启停、导出。

## 9. 关键业务规则

### 9.1 渠道规则
- 学生与渠道方为一对一有效绑定关系。
- 订单归属主体始终为学生本人，不因渠道可见或渠道代付而改变。
- 若学生在支付时存在有效绑定渠道方，则该订单进入对应渠道统计与返点口径。
- 历史渠道统计与返点归属按支付发生时的有效绑定渠道锁定，不因后续更换绑定而变化。
- 渠道支持为未注册参赛人创建临时参赛档案，并用于批量报名。
- 付费团队预约必须拆分为“1 条团队预约主记录 + N 条成员预约明细 + N 笔个人子订单”，团队主记录不是支付主体。
- 每笔成员子订单仍归属对应学生；未注册成员则归属对应 `participant_id`。
- 渠道可一次性合并支付多笔成员子订单，但账务、退款、返点、结算仍按子订单逐笔处理。
- 返点规则按“赛事 + 渠道方”配置。

### 9.2 金额规则
- 用户净实付 = 订单标价金额 - 优惠券抵扣 - 积分抵扣。
- 优惠券与积分抵扣默认由平台承担。
- 默认返点基数 = `max(0, 订单标价金额 - 平台服务费)`。
- 平台服务费支持后台灵活配置。
- 零元订单仅指标价金额 = 0 的免费 / 公益订单；该类订单不发放积分、成长值、渠道返点。
- 全额抵扣订单指标价金额 > 0、但优惠券 / 积分抵扣后用户净实付 = 0 的订单；该类订单正常发放积分、成长值，并参与渠道返点。
- 所有零付订单（零元订单 + 全额抵扣订单）正常占用赛事名额、线下预约名额和商城库存，不得绕过资源占用逻辑。
- 全额抵扣订单仍需进入赛事方结算口径，且结算单需展示优惠明细、用户净实付、平台承担金额。
- 售后截止时间默认取 `max(赛事结束时间, 成绩发布时间) + 7` 个自然日；截止后订单自动转为 `completed`。

### 9.3 积分规则
- `100 积分 = 1 元`
- 单笔报名积分抵扣不设上限。
- 积分永不过期。
- 成长值决定等级，积分用于抵扣与兑换。
- 零元订单不发放积分与成长值。
- 全额抵扣订单正常发放积分与成长值。

### 9.4 商城规则
- 实物商品需支持库存、地址、发货、物流、售后。
- 售后退款通过后需联动回退积分、现金与库存。

### 9.5 赛事方规则
- 赛事方结算方式按赛事一事一议。
- 平台保留赛事方入驻审核权、赛事审核权、成绩证书审核权、最终下架权。
- 赛事方结算明细需可拆分展示：标价金额、优惠券金额、积分金额、平台承担优惠、用户实付、退款、渠道返点、平台服务费。
- 标价金额 > 0 的全额抵扣订单，仍按有价订单进入赛事方结算与渠道返点口径。
- 赛事结算仅在订单已过售后截止且无进行中售后时开放生成。

### 9.6 优惠券与营销规则
- 默认单笔订单最多使用 1 张优惠券，且可与积分同时使用。
- 用户无可用优惠券时，前端不展示优惠券模块。
- 优惠券核销顺序先于积分。
- 活动专题、优惠券模板、发券批次必须可追踪到活动来源与操作人。
- 退款后需按规则执行优惠券回退或失效回退。
- 退款资产回退优先级固定为：现金 -> 积分 -> 优惠券；部分退款默认不回退优惠券；整单全退仅在原券有效期内时回退原券。

### 9.7 权益核销规则
- 权益必须有来源、额度 / 次数、有效期、核销方式、回退规则。
- 核销通过后需进入权益台账，并可与结算或活动对账关联。

### 9.8 风控规则
- 风险规则、预警、工单、黑白名单四类对象必须打通。
- 风控必须覆盖所有零付订单（零元订单 + 全额抵扣订单）的频次、设备、手机号、渠道维度拦截。
- 风险工单关闭前必须记录处理结论与复核人。

### 9.9 合同与开票规则
- 合同与结算单、提现单、发票记录必须可关联追溯。
- 开票状态、红冲状态、快递状态需可回写到合作方档案与财务记录。

### 9.10 线下预约与核销规则
- 线下活动支持预热发布、预约开放、预约关闭、活动结束、取消等状态。
- 当前版本预约粒度默认按“日期 + 场次/时段”配置。
- 是否允许重复预约由后台按活动配置。
- 预约主体支持个人预约和团队预约。
- 团队预约必须提交成员名单，并支持填写额外人数与额外名单，用于家长、引路工作人员等非正式成员。
- 一个预约主记录可挂接多个核销子项二维码。
- 付费团队预约不生成单笔团队总订单，而是按成员名单生成成员预约明细与个人子订单，再归集到团队主记录。
- 额外人数与额外名单默认仅用于容量统计、现场识别与核销展示，不生成学生子订单，不参与返点与财务结算。
- 核销子项至少支持：到场、礼品、颁奖典礼、自定义。
- 每个子项独立核销、独立过期、独立作废、独立日志。
- 当前版本礼品核销不联动库存。
- 现场核销终端固定使用 `event-writeoff.html`，扫码后自动区分个人预约和团队预约。
- 团队预约详情需展示渠道名称、团队名称、预约总人数、成员人数、额外人数、已核销人数。
- 团队预约核销需支持整单核销和按人数部分核销，且允许跨多次累计完成。
- 场次结束后：0 人核销记为 `no_show`；部分核销记为 `partial_writeoff_ended`；全部核销维持 `fully_writeoff`。
- 团队预约相关现场履约、资源消耗、运营对账默认按实际已核销人数统计。
- 当前版本团队预约支付金额按预约锁定人数成立，不因部分未到场自动减免。
- 场次结束后的未核销人数默认不自动退款；如需补退，仅允许后台人工发起部分退款，并同步回滚结算与返点。
- 与团队预约关联的有价订单默认按预约锁定人数进入财务结算，除非合同明确约定按实际核销人数结算。
- 现场核销记录必须保留核销人员、核销终端、核销时间和核销结果。

### 9.11 报名动态表单与临时参赛档案规则
- 每个赛事报名表单按模板驱动，前后端都不得写死字段。
- 表单模板变更后不回写历史报名数据。
- 临时参赛档案支持后续认领合并到正式账号。
- 当前版本自动认领规则默认按“监护人手机号 + 参赛人姓名”完全匹配。
- 仅手机号命中或存在多候选时，不自动认领，进入后台人工确认池。
- 合并时底层交易流水保留原 `participant_id` 快照，通过 `participant_user_map` 映射表归集到正式 `user_id`。
- 因可能存在多个学生共用同一手机号，后台必须支持人工确认认领、改绑和撤销误合并。
- 赛事报名唯一编号可作为辅助检索字段，但不作为唯一认领键。

## 10. 非功能需求

- 列表页默认支持分页、排序、筛选、导出。
- 金额字段统一两位小数，支持千分位展示。
- 关键配置页面支持操作日志查询。
- 导出数据需受权限控制。
- 资金相关操作需支持二次确认。
- 打款、积分调整、库存调整、售后退款等关键动作必须写操作日志。
- 风控处置、合同签署、开票流转、优惠券发放、权益回退必须写操作日志。
- 后台登录、菜单、按钮权限、数据权限采用统一后台安全体系。
- 页面需适配 1440 宽及以上主流后台办公分辨率。

## 11. 开发落地规范

### 11.1 前端规范
- 基于独立后台前端工程新增竞赛通业务菜单与页面。
- 表格页统一采用查询栏、工具栏、导出、字典标签的标准后台模式。
- 复杂详情页采用“列表页 + 抽屉 / 弹窗详情”与“详情页独立路由”结合。

### 11.2 后端规范
- 业务领域按模块拆分：用户、渠道、赛事、订单、营销、权益、积分、商城、财务、合同、风控、AI。
- 关键流水表需独立设计：积分流水、成长值流水、优惠券流水、权益台账、返点台账、提现单、赛事结算单、打款记录、售后单、风险工单。
- 临时参赛档案与正式账号必须通过独立映射表关联，不得直接覆盖历史交易流水中的 `participant_id`。
- 动态报名表单主存储采用 JSON / JSONB 快照模式，导出层按模板版本解析，不采用通用 EAV 作为主模型。
- 团队预约需独立设计 `team_appointment`、`team_appointment_member` 等聚合表；订单层仍使用成员级个人子订单，不得以团队主记录替代订单主体。
- 所有审核类接口需幂等设计。

### 11.3 后台权限落地规范
- 菜单通过后台权限配置中心维护。
- 角色通过角色管理中心维护。
- 页面按钮通过 `perms` 控制。
- 数据权限在业务 SQL 层或服务层实现。
