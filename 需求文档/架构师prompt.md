**【技术栈与基建约束】（严格遵守）**

**【数据库技术栈与建表约束】（极度重要）**
1. **版本锁定**：必须严格采用 **MySQL 5.7** 的 DDL 语法规范。不要使用 MySQL 8.0 或 PostgreSQL 的专有特性。
2. **建表规范**：
    - 存储引擎统一使用 `InnoDB`。
    - 字符集统一使用 `utf8mb4`，排序规则使用 `utf8mb4_general_ci`（防止 Emoji 表情导致报错）。
    - 主键尽量采用业务无关的雪花算法 ID（`bigint`）或自增 ID。
3. **若依兼容性（核心）**：每个建表语句（`CREATE TABLE`）以及每一个字段，**必须包含详尽的中文 `COMMENT` 注释**。这是若依代码生成器识别字段含义的唯一依据。请自带 `create_time`, `update_time`, `create_by`, `update_by`, `remark` 等若依标准审计字段。
4. **JSON 存储边界**：PRD 中提到的动态表单快照，请使用 MySQL 5.7 原生的 `JSON` 数据类型，不要写成 `JSONB`。
5. **不要重复造轮子**：系统基础表（如 `sys_user`, `sys_role`, `sys_dict_data` 等）是若依自带的，**请绝对不要输出这些基础表的 DDL**。你只需要输出 PRD 中新增的“业务表”（如：临时档案映射表、团队订单表、返点台账表、赛事配置表等）的完整 DDL 语句。

**【服务端架构与代码编写规范】（RuoYi 深度适配）**
1. **架构分层体系**：严格遵守 `Controller -> Service -> Mapper(MyBatis)` 的经典三层架构。针对 PRD 中的核心业务，请按领域（Domain）划分为不同模块（如 `jst-system`, `jst-order`, `jst-points` 等），不要全部堆在 `ruoyi-admin` 模块下。
2. **RuoYi 核心类复用**：
    - 所有的 Controller 必须继承 `BaseController`。
    - 接口返回统一使用 `AjaxResult`（如 `AjaxResult.success()`）。
    - 分页查询必须基于若依原生的 `startPage()` 和 `getDataTable()` 封装。
3. **权限与安全约束**：
    - 方法级权限控制必须使用若依原生注解，严格对应 PRD 中的权限标识。例如：`@PreAuthorize("@ss.hasPermi('jst:rebateRule:list')")`。
    - 数据传递必须进行**严格的对象隔离**：接收参数必须使用 `ReqDTO` 或 `Form`（带 JSR-303 校验注解，如 `@NotBlank`），返回前端必须使用 `ResVO`。**绝对禁止直接返回数据库 Entity 实体类**。
4. **事务与并发防雷（核心）**：
    - 针对 PRD 中提到的“退款资金流转”、“订单状态流转”、“团队预约名额扣减”、“返点计提与负向抵扣”，服务层方法必须添加 `@Transactional(rollbackFor = Exception.class)`。
    - 针对高并发可能引发超卖的场景（如：团队部分核销、商城库存扣减），请在架构设计中指明必须引入 Redis 分布式锁（如 Redisson）或数据库乐观锁。
