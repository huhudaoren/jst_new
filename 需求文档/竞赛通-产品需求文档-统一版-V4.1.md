## 3. 当前产品定义

### 3.1 产品定位

竞赛通当前定位为以**微信小程序**为核心用户端、以**赛事方 Web 工作台**为供给端、以**平台 Web + H5**为管理端的竞赛服务平台，核心赛道为文化艺术类赛事，核心业务闭环为：

`找赛事 -> 绑定渠道 -> 报名/代报名 -> 优惠券/积分抵扣 -> 支付 -> 线下预约/现场核销 -> 审核 -> 成绩/证书 -> 积分/等级 -> 商城兑换 / 权益核销 -> 返点/结算 / 合同开票 -> AI 学习`

### 3.2 当前业务目标

- 建立统一的文化艺术赛事展示、报名、审核、成绩、证书、课程服务闭环。
- 建立渠道激励与返点结算体系，支撑渠道招生、代报名、提现和财务对账。
- 建立积分、成长值、等级、优惠券、权益、积分商城等用户运营体系。
- 建立第三方赛事方入驻、发赛、管赛、导入成绩、发证、结算的供给侧平台能力。
- 建立线下活动“预热发布、日期预约、现场多码核销”的完整闭环。
- 保留 OpenMAIC AI 课程差异化能力，形成“赛事 + 学习 + 运营”一体化模式。
- 建立风控、合同、开票、经营分析等平台化运营能力。

### 3.3 当前建设边界

- 主业务端：微信小程序。
- 赛事方工作端：Web 工作台。
- 平台高频审核端：H5。
- 平台复杂配置端：Web 后台。
- 当前不做：多级分销返佣、积分转赠/提现、完整线上场地租赁交易、社区互动、地区分站。

## 4. 角色与终端

### 4.1 终端与角色映射

| 终端 | 主要角色 | 主要能力 |
|------|------|------|
| 微信小程序 | 学生、家长、渠道方 | 找赛事、报名、支付、积分、优惠券、权益、商城、消息、AI 学习、渠道工作台 |
| 赛事方公开申请页 | 赛事方申请人 | 入驻申请、补充材料、查看审核进度 |
| 赛事方 Web 工作台 | 赛事方 | 赛事管理、报名管理、成绩导入、证书管理、赛事结算、合同/开票查看 |
| 平台 H5 审核端 | 平台管理员 | 高频审核、发布、确认、打款状态标记、风险确认 |
| 平台 Web 后台 | 平台管理员、运营、财务 | 复杂配置、规则管理、报表、对账、批量导入导出、合同与票据管理 |

### 4.2 角色定义

- **学生**：浏览赛事、自主报名、查询进度、查询成绩、下载证书、获得积分、参与兑换、学习普通课程与 AI 课程。
- **家长**：协助学生完成报名、支付、资料上传、地址维护、售后沟通。
- **渠道方**：统一业务角色，包含三种认证类型：
  - 老师渠道方
  - 机构渠道方
  - 个人渠道方
- **赛事方**：入驻的第三方赛事公司，拥有独立 Web 工作台。
- **现场核销人员**：赛事方或平台授权的现场执行身份，负责扫码完成到场、礼品、典礼等现场核销；不是独立业务结算主体。
- **平台管理员**：负责审核、发布、结算、风控确认和平台运营。

### 4.3 角色关系基线

- 学生与渠道方是**一对一有效绑定关系**，同一时间只能存在一个有效绑定。
- 学生与赛事方**不建立绑定关系**，学生仅报名赛事方发布的赛事。
- 渠道方不再存在“机构管理多个老师”的层级模型；老师、机构、个人只是**渠道认证类型不同**，不是组织上下级。
- 赛事方不参与积分等级体系；积分等级体系只面向学生与渠道方。
- 现场核销人员只是赛事方或平台的操作身份，不构成新的业务角色模型。

## 5. 核心业务闭环

### 5.1 学生自主报名、渠道可见与积分获得

```text
首页 / 搜索
-> 赛事列表
-> 赛事详情
-> 登录
-> 报名流程（填资料 -> 上传作品 -> 支付）
-> 订单归属学生本人
-> 若学生支付时存在有效绑定渠道方，则订单同步进入该渠道方统计视图
-> 支付成功
-> 发放积分与成长值
-> 报名审核
-> 我的报名 / 我的成绩 / 我的证书
```

### 5.2 渠道绑定、代报名与返点

```text
渠道方分享绑定链接 / 二维码
-> 学生进入 binding.html
-> 建立一对一有效绑定
-> 渠道方查看已绑定学生
-> 学生本人下单后，订单归属仍为学生本人
-> 若学生存在未支付订单，渠道方可查看并代为支付
-> 渠道方支付成功后，学生侧显示“渠道方已缴费”，学生本人不可重复支付
-> 支付完成时锁定“支付时有效绑定渠道方”
-> 售后截止后，若订单状态为 `completed` 且订单标价金额 > 0
-> 按“赛事 + 渠道方”规则计提返点并纳入渠道统计
-> 渠道方发起提现
-> 平台审核 / 打款
-> 渠道方查看结算单、发票、打款状态
```

### 5.3 积分、成长值与等级

```text
触发行为（报名 / 购课 / 签到 / 邀请 / 学习完成 / 获奖）
-> 系统发放积分与成长值
-> 成长值累计达到阈值
-> 等级升级
-> 新权益生效
-> 用户在 points-center.html 查看等级、积分与权益
```

### 5.4 抵扣、优惠、兑换与售后

```text
用户下单
-> 选择优惠券
-> 勾选积分抵扣
-> 计算净实付
-> 若应付 > 0 则继续支付
-> 若应付 = 0 且标价金额 = 0，则生成零元订单并进入审核
-> 若应付 = 0 且标价金额 > 0，则生成全额抵扣订单并进入审核
-> 零元订单不发放积分、成长值和渠道返点；全额抵扣订单按正常有价订单处理

用户进入积分商城
-> 浏览商品
-> 兑换商品
-> 查看物流 / 售后
-> 按规则回退积分与现金
```

### 5.5 赛事方入驻、发赛、赛后结算

```text
赛事方提交入驻申请
-> 平台审核通过
-> 开通赛事方账号
-> 创建赛事并提交审核
-> 平台审核上架
-> 开放报名
-> 赛后导入成绩 / 生成证书
-> 平台审核发布
-> 生成赛事结算单
-> 赛事方确认结算
-> 平台审核打款
```

### 5.6 线下活动预约与现场多码核销

```text
线下活动提前发布预热页
-> 用户查看活动信息、预约开放时间、可预约日期
-> 预约开放后进入 appointment-select.html 选择预约日期（按年 / 月 / 日）
-> 提交预约
-> 系统生成预约记录与多个核销子项二维码
-> 用户在 appointment-detail.html 查看到场码 / 礼品码 / 典礼码等独立二维码
-> 现场核销人员使用 event-writeoff.html 扫码核销对应子码
-> 单个子码独立核销，不影响其他子码
-> 用户在 my-appointment.html 查看预约状态、部分核销状态与全部核销结果
```

### 5.7 AI 课程创建与学习

```text
已认证渠道方或平台管理员进入 AI 课程模块
-> 选择主题 / 上传素材
-> 调用 OpenMAIC 生成课件
-> 预览、编辑、发布
-> 平台审核
-> 学生在 my-course.html 进入 AI 课堂学习
-> 回传学习进度、测验结果
-> 发放积分与成长值
```

### 5.8 平台 H5 高频审核

```text
admin-login.html
-> admin.html 工作台
-> 处理认证、报名、凭证、成绩、赛事方、渠道提现、赛事结算、风险确认
-> 写入审计日志
-> 通知相关用户
```

## 7. 关键功能要求

### 7.1 赛事与报名

- 赛事列表需支持来源标记：平台自营 / 赛事方入驻。
- 赛事列表需支持筛选：赛事分类、是否支持积分抵扣、是否支持渠道报名。
- 赛事详情需展示：赛事基础信息、来源、报名时间、费用、奖项、审核状态。
- 若赛事支持渠道报名，需明确展示“支持渠道方统一代报名”说明。
- 每个赛事必须支持**独立报名表单模板**，不得假设所有赛事提交字段一致。
- 报名流程至少三步：填写资料、支付、完成。
- 支付方式支持：微信支付、对公转账、积分抵扣、积分 + 现金混合支付。
- 零元订单允许正常创建，并进入标准审核与通知流程。
- 学生本人创建的订单，归属主体始终为学生本人，不因渠道可见或渠道代付而变更归属主体。
- 若学生在支付时存在有效绑定渠道方，则该订单必须进入该渠道方统计视图，并可参与该渠道方返点计算。
- 若学生本人存在待支付订单，已绑定渠道方可在渠道视角代为完成支付；支付完成后学生侧显示“渠道方已缴费”，学生不可再次支付。
- 若为团队预约付费场景，系统必须拆分为“1 条团队预约主记录 + N 条成员预约明细 + N 笔个人子订单”；团队主记录不是订单主体。
- 订单详情需展示：报名信息、作品/证件、支付方式、支付发起方、优惠券金额、积分抵扣金额、平台承担优惠金额、净实付、归属学生、关联渠道方、时间轴。

#### 动态报名表单要求

- 报名表单采用**赛事表单模板驱动**，模板由平台后台或赛事方工作台配置，经平台审核后生效。
- 字段类型至少支持：
  - 单行文本
  - 多行文本
  - 单选
  - 多选
  - 下拉选择
  - 日期
  - 年龄
  - 数字
  - 图片上传
  - 音频上传
  - 视频上传
  - 通用文件上传
  - 分组字段
  - 条件显示字段
- 字段级配置至少支持：
  - 字段编码
  - 字段名称
  - 字段说明
  - 是否必填
  - 校验规则
  - 可上传文件类型
  - 文件大小限制
  - 文件数量限制
  - 展示顺序
  - 条件显示逻辑
- 示例：
  - 安徒生类赛事可包含：姓名、作品图、组别、年龄、分组、指导老师、作品说明。
  - 录音类赛事可包含：姓名、音频文件、视频文件、Word/PDF 附件、曲目说明、时长。
- 前端必须按模板动态渲染，后端必须按模板动态校验，不能写死字段。
- 表单模板变更后，不影响历史已提交报名单的字段快照。

### 7.2 渠道方模块

- 渠道认证类型仅保留：老师 / 机构 / 个人。
- 认证通过后才开放渠道视角、绑定码、代报名、返点、提现、等级权益。
- 渠道工作台需展示：绑定学生数、代报名数、待结算返点、累计返点、等级权益。
- 学生管理需支持：查看已绑定学生、查看历史绑定记录、为学生代报名、查看成绩、查看证书。
- 渠道订单页需支持：按学生、赛事、时间、支付状态、退款状态筛选。
- 渠道数据页定位为轻量运营页，不承担深度经营分析。

#### 渠道批量报名与未注册参赛人

- 渠道必须支持**批量报名**，用于学校、机构、培训班集中代报名。
- 批量报名入口放在 `channel-batch-enroll.html`。
- 批量报名对象分两类：
  - 已绑定且已注册学生
  - 未注册参赛人
- 对于未注册参赛人，系统必须支持创建**临时参赛档案**：
  - 有 `participant_id`
  - 可无 `user_id`
  - 可直接用于报名、上传作品、支付、出成绩、生成证书
- 临时参赛档案至少包含：
  - 姓名
  - 性别（如赛事需要）
  - 出生日期 / 年龄（如赛事需要）
  - 监护人姓名（如赛事需要）
  - 监护人手机号
  - 学校 / 机构 / 班级（如赛事需要）
  - 证件信息（如赛事需要）
  - 渠道创建人
- 渠道批量报名方式至少支持：
  - 手动逐个新增
  - Excel 模板导入基础信息
  - 按模板批量上传附件
- 对于未注册参赛人：
  - 报名期间由渠道方管理
  - 成绩、证书、订单、预约、核销均先挂在临时参赛档案下
  - 后续如学生完成注册，系统需支持将临时参赛档案与正式账号合并认领
  - 自动认领规则默认按“监护人手机号 + 参赛人姓名”同时完全匹配
  - 若仅手机号命中但姓名不一致，则进入人工待认领，不允许系统自动并档
  - 合并时底层交易流水保留原 `participant_id` 快照，通过映射关系将资产归集到正式 `user_id`
  - 因存在多个学生共用同一手机号的情况，系统必须支持平台管理员人工确认、改绑、撤销误合并
  - 赛事报名唯一编号可作为辅助检索字段，但当前版本不作为唯一认领键
- 临时参赛档案在未认领前，不进入普通学生公共列表，只对创建渠道、平台管理员和相关赛事方可见。

### 7.3 返点与结算

- 返点按“赛事 + 渠道方”配置，不再按机构层级或多级分销配置。
- 返点支持：比例返点、固定金额返点。
- 订单归属主体与返点归属主体分离：
  - 订单归属主体固定为学生
  - 返点与统计归属按“支付时有效绑定渠道方”锁定
- 返点中心需展示：可提现余额、审核中金额、累计打款、退款回退。
- 返点明细至少包含：赛事、学生、订单号、净实付、平台服务费、返点规则、返点金额、赛事结束时间。
- 结算单页需支持：勾选提现项、填写发票信息、确认收款账户、查看打款凭证和驳回原因。

### 7.4 积分、成长值与等级

- 系统维护两套资产：
  - **成长值**：用于升级，默认只增不减，可后台人工调整。
  - **积分**：用于抵扣、兑换、售后回滚，可补发、扣减、冻结、解冻。
- 行为来源至少包括：报名、购课、签到、邀请、学习完成、获奖。
- 零元订单不发放积分、成长值。
- 同一套等级引擎服务学生与渠道方，但前端展示权益内容允许按角色差异化。
- 积分中心需展示：当前等级、成长值进度、可用积分、冻结积分、累计获取积分、赚积分任务。

### 7.5 优惠券、权益与积分商城

- 优惠券来源支持：平台发放、活动发放、后台补发。
- 优惠券类型支持：满减券、直减券、折扣券、指定赛事券。
- 单笔订单默认最多使用 **1 张优惠券**，并可与积分同时使用。
- 下单页需支持：自动推荐最优优惠券、手动切换、展示净实付变化。
- 若当前用户没有任何可用优惠券，则下单页不展示优惠券模块及优惠信息区。
- 权益中心需支持：可用权益、待核销权益、已失效权益。
- 权益核销需支持：线上自动核销、人工审核核销、线下确认核销。
- 积分商城支持两类商品：虚拟商品、实物商品。
- 兑换订单需支持：地址、库存、发货状态、物流单号、售后、退款回退。

### 7.6 赛事方入驻与工作台

- 赛事方入驻申请页需支持草稿、提交审核、查看审核状态、补充资料重提。
- 赛事方工作台需支持：赛事管理、报名管理、成绩导入 / 发布、证书管理、赛事结算。
- 赛事创建页需支持配置：是否支持渠道代报名、是否支持积分抵扣、证书规则、成绩规则、是否支持线下预约、预约日期配置、核销子项配置。
- 成绩与证书发布必须经过平台审核后才可对外生效。
- 赛事方结算页需展示：订单标价金额、优惠券金额、积分抵扣金额、用户净实付、平台承担优惠金额、渠道返点、平台服务费、退款、合同约定扣项、最终结算金额。
- 赛事方现场执行需支持专用 `event-writeoff.html` 页面，用于扫码核销预约子码。

### 7.7 线下预约与现场多码核销

- 线下活动支持**预热发布**：活动可在预约未开始前提前对外展示。
- 线下活动预约按**日期 + 场次 / 时段**配置。
- 用户预约时需在 `appointment-select.html` 中依次选择**年 / 月 / 日 + 场次 / 时段**。
- 是否允许同一用户重复预约，由后台按活动配置：
  - 允许重复预约
  - 禁止重复预约
- 每个预约记录对应 1 条预约主记录，可挂接多个独立核销子项。
- 预约类型分为：
  - 个人预约
  - 渠道团队预约
- 核销子项默认支持以下类型，且允许按活动配置启停：
  - 到场核销
  - 礼品核销
  - 颁奖典礼核销
  - 自定义核销项
- 每个核销子项拥有独立二维码、独立状态、独立核销时间和核销人。
- 任一子码核销成功后，仅更新该子项状态，不自动核销其他子项。
- 当前版本**不引入礼品库存联动**，礼品核销只做资格核销与记录，不做库存扣减。
- `my-appointment.html` 需展示预约列表、预约日期、活动状态、整体核销进度。
- `appointment-detail.html` 需展示：活动信息、预约日期、主预约状态、多码二维码卡片、各子项状态和核销时间。
- `channel-appointment.html` 需支持渠道批量预约：
  - 选择活动
  - 选择日期与场次
  - 填写成员名单
  - 填写额外人数与额外名单（如家长、引路工作人员）
  - 系统自动汇总预约总人数
  - 生成团队预约记录
  - 若活动涉及付费，按成员名单同步生成成员预约明细与个人子订单
- 渠道团队预约需展示：
  - 渠道名称
  - 团队名称或批次名称
  - 预约总人数
  - 成员人数
  - 额外人数与额外名单
  - 已核销人数
  - 剩余可核销人数
  - 预约日期与场次
- 团队预约底层模型必须满足：
  - `team_appointment` 仅作为团队聚合主记录，不承载资金归属
  - `team_appointment_member` 逐个记录成员预约明细
  - 每个正式成员生成 1 笔个人子订单；未注册成员则生成归属到 `participant_id` 的个人子订单
  - 渠道可一次性合并支付多笔个人子订单，但不改变子订单归属主体
  - 额外人数与额外名单默认仅用于容量统计、现场识别与核销展示，不生成学生子订单，不参与返点与财务结算
- `event-writeoff.html` 需支持扫码后自动区分：
  - 个人预约：显示个人信息
  - 团队预约：显示团队预约信息
- 个人预约扫码结果至少展示：姓名、手机号、预约日期、预约场次、核销子项、当前状态。
- 团队预约扫码结果至少展示：渠道名称、团队名称、预约总人数、成员人数、额外人数、已核销人数、预约日期、预约场次、核销子项、当前状态。
- 团队预约核销默认支持“整单核销”或“按人数部分核销”，并允许同一团队码跨多次累计核销，直到预约人数用完或活动结束。
- 当前版本团队预约支付金额按预约锁定人数成立，场次结束后未核销人数不自动触发退款。
- 如需对未到场人数补退，仅允许平台管理员后台发起人工部分退款，并同步回滚结算与返点。
- 预约主状态至少包括：待预约 / 已预约 / 部分核销 / 全部核销 / 部分核销已结束 / 已取消 / 已过期 / 已失约。
- 核销子项状态至少包括：未核销 / 已核销 / 已过期 / 已作废。

### 7.8 OpenMAIC AI 课程

- OpenMAIC 继续采用独立部署 + API 调用 + SSO 集成。
- AI 课程创建权限仅开放给：
  - 已认证渠道方
  - 平台管理员
- 学生、家长、赛事方不具备 AI 课程创建权限。
- 小程序端允许“大文件上传由 Web 承接页完成”，不用强行全部原生化。
- AI 学习完成可触发积分和成长值奖励。
- AI 课程权益可纳入等级权益体系。

### 7.9 个人中心、消息与账号

- `my.html` 保留双视角：学生视角 / 渠道方视角。
- 若用户同时具备学生与渠道方身份，可切换视角，但各自订单和资产必须按身份隔离显示。
- `my-appointment.html` 作为学生侧线下预约主入口，挂在“我的”或“我的服务”中。
- `notice-msg.html` 作为公共公告 / 系统消息中心。
- `my-message.html` 作为订单、售后、物流、积分、权益、提现、结算等个人服务消息中心。
- `settings.html` 负责账号安全、地址入口、通用设置；地址子页可作为下级页面存在。

### 7.10 平台 H5 审核端与 PC 后台边界

- H5 管理员端聚焦：审核、确认、发布、打款状态标记、风险确认。
- PC 后台聚焦：复杂规则配置、批量导入导出、合同与票据维护、报表与对账。
- H5 管理员至少覆盖 7 类任务：认证、报名、凭证、成绩、赛事方、渠道提现、赛事结算。
- 积分规则、等级规则、权益配置、商城商品、返点规则、平台服务费规则等复杂配置不放入 H5 管理员端。

## 8. 关键规则与数据口径

### 8.1 基础字典

| 项目 | 规则 |
|------|------|
| 证书编号格式 | `JST-YYYY-ART-XXXXX` |
| 赛事分类 | 艺术 / 音乐 / 舞蹈 / 美术 / 朗诵/戏剧 / 文化 / 科技 / 体育 |
| 参赛组别 | 幼儿园组 / 小学低年级组 / 小学高年级组 / 初中组 / 高中组 |
| 奖项等级 | 国际金奖 / 国际银奖 / 全国金奖 / 全国银奖 / 全国铜奖 / 优秀奖 / 参与证书 |

### 8.2 学生与渠道方关系

| 数据项 | 规则 |
|------|------|
| 有效绑定数 | 同一时间仅 1 条有效绑定 |
| 绑定对象 | 渠道方（老师 / 机构 / 个人） |
| 更换绑定 | 新绑定生效后旧绑定失效 |
| 历史渠道关联 | 以支付发生时的有效绑定渠道锁定，不因后续更换绑定回溯变更 |
| 学生与赛事方 | 不建立绑定关系 |

### 8.3 临时参赛档案规则

| 数据项 | 规则 |
|------|------|
| 档案类型 | `registered_participant / temporary_participant` |
| 主键 | 已注册用 `user_id`，未注册用 `participant_id` |
| 创建主体 | 渠道方 / 平台管理员 |
| 可承载业务 | 报名、支付、作品上传、成绩、证书、预约、核销 |
| 可见范围 | 创建渠道、平台管理员、相关赛事方 |
| 后续处理 | 注册后支持认领并合并到正式账号 |
| 自动认领规则 | 默认按“监护人手机号 + 参赛人姓名”同时完全匹配 |
| 待人工认领 | 仅手机号命中或存在多候选时，不自动认领，进入人工确认池 |
| 合并模型 | 通过 `participant_user_map` 映射表归集资产，不直接覆盖原始交易流水主键 |
| 历史流水处理 | 订单、成绩、证书、预约、核销、作品等底层流水保留 `participant_id` 快照 |
| 人工纠错 | 支持平台管理员人工确认、改绑、撤销误合并 |
| 辅助识别字段 | 赛事报名唯一编号可辅助检索，但不作为唯一认领键 |

### 8.4 订单归属、支付与抵扣

| 数据项 | 当前基线 |
|------|------|
| 订单归属主体 | 始终归属学生本人 |
| 订单可见范围 | 学生本人必可见；若支付时存在有效绑定渠道方，则该渠道方可见 |
| 渠道统计归属 | 按支付时有效绑定渠道方锁定 |
| 渠道代付 | 支持渠道为学生待支付订单完成支付 |
| 团队预约订单模型 | 付费团队预约必须拆分为 `1 条团队预约主记录 + N 条成员预约明细 + N 笔个人子订单` |
| 团队主记录角色 | 仅作为预约聚合容器，不作为支付、退款、返点、结算主体 |
| 团队子订单归属 | 每笔成员子订单仍归属对应学生；未注册成员归属对应 `participant_id` |
| 团队合并支付 | 渠道可对多笔成员子订单一次性合并支付，但账务仍按子订单逐笔入账 |
| 支付方式 | 微信 / 对公转账 / 积分抵扣 / 积分+现金 |
| 积分比例 | `100 积分 = 1 元` |
| 积分抵扣上限 | 当前文稿定义为不设上限 |
| 零元订单定义 | 仅指订单标价金额本身 = 0 的免费 / 公益订单 |
| 全额抵扣订单定义 | 订单标价金额 > 0，但因优惠券 / 积分抵扣后用户净实付 = 0 |
| 零付订单定义 | 用户净实付 = 0 的订单统称，包含零元订单与全额抵扣订单 |
| 零元订单 | 允许创建并进入审核流程 |
| 零元订单奖励 | 不发放积分、成长值、渠道返点 |
| 全额抵扣订单奖励 | 正常发放积分、成长值，并按有价订单参与渠道返点计提 |
| 零付订单资源占用 | 正常占用赛事名额、线下预约名额、商城库存 |
| 退款资产顺序 | 默认先退现金，再退积分，最后回退优惠券 |
| 部分退款规则 | 部分退款优先退现金；现金不足时再退积分；优惠券默认不因部分退款回退 |
| 全额退款规则 | 先退现金，再退积分；优惠券仅在整单全退且退款完成时原券仍在有效期内时退回，已过期默认不补发 |
| 售后截止时间 | 默认取 `max(赛事结束时间, 成绩发布时间) + 7` 个自然日，可按赛事配置覆盖 |
| 订单完结状态 | 售后截止后且无进行中售后时，订单自动转为 `completed` |
| 用户退款互斥 | 任一核销子码已核销或证书已发放后，用户侧禁止自主发起退款 |
| 后台特批退款 | 仅平台管理员可执行特批退款，且必须同步作废证书、核销资格并留痕 |
| 订单支付后学生侧展示 | 若由渠道代付，则显示“渠道方已缴费” |

### 8.5 返点与平台服务费

| 数据项 | 规则 |
|------|------|
| 配置颗粒度 | `赛事 + 渠道方` |
| 返点模式 | `rate`（比例）/ `fixed`（固定金额） |
| 优惠券承担方 | 平台 |
| 积分抵扣承担方 | 平台 |
| 用户净实付 | `订单标价金额 - 优惠券金额 - 积分抵扣金额` |
| 返点基准金额 | 默认按 `订单标价金额 - 平台服务费` 计算，不因平台承担的优惠券与积分抵扣而减少 |
| 适用订单范围 | 仅排除“标价金额 = 0”的零元订单；全额抵扣订单按正常有价订单参与返点 |
| 核算粒度 | `order_item` 明细级分摊与回滚 |
| 计提时点 | 售后截止后、订单状态为 `completed`、订单标价金额 > 0 时方可计提 |
| 提现状态 | `pending / reviewing / rejected / approved / paid` |
| 回退规则 | 已计提未打款原路扣回；已打款进入负向台账 |
| 负向台账处理 | 渠道提现审核时系统自动优先抵扣负向台账；不足部分结转至后续批次继续抵扣 |

### 8.6 积分、成长值与等级

| 数据项 | 规则 |
|------|------|
| 参与角色 | `student` / `channel` |
| 赛事方 | 不参与积分等级体系 |
| 成长值 | 决定等级，默认不失效 |
| 积分 | 用于抵扣、兑换、售后回滚 |
| 积分状态 | 可用 / 冻结 / 已使用 |
| 流水类型 | 获取 / 扣减 / 冻结 / 解冻 / 调整 / 回退 |
| 配置方式 | 等级数量、阈值、图标、权益后台可配置 |
| 零元订单 | 不发放积分与成长值 |
| 全额抵扣订单 | 正常发放积分与成长值 |

### 8.7 优惠券、权益、商城

| 数据项 | 规则 |
|------|------|
| 优惠券类型 | 满减 / 直减 / 折扣 / 指定赛事券 |
| 发放来源 | 平台发放 / 活动发放 / 后台补发 |
| 订单用券数量 | 默认单笔订单最多 1 张 |
| 核销顺序 | 默认先优惠券，后积分 |
| 优惠信息展示 | 用户无可用优惠券时，不展示优惠券模块 |
| 优惠券退款回退 | 默认仅整单全额退款且退款完成时原券仍在有效期内时回退原券；部分退款不回退优惠券 |
| 权益来源 | 等级权益 / 活动权益 / 后台补发 |
| 权益口径 | 固定次数 / 固定金额 / 固定有效期 |
| 商城商品类型 | `virtual` / `physical` |
| 售后类型 | 退款 / 退货退款 |

### 8.8 赛事方结算

| 数据项 | 规则 |
|------|------|
| 结算方式 | 一事一议，按赛事合同配置 |
| 默认口径 | `最终结算金额 = 订单标价金额合计 - 渠道返点 - 平台服务费 - 退款 - 约定扣项` |
| 结算触发时点 | 售后截止后且无进行中售后时，方可生成结算单 |
| 发起主体 | 赛事方确认 / 平台审核 |
| 结算状态 | 待确认 / 审核中 / 已驳回 / 待打款 / 已打款 |
| 平台权力 | 保留最终审核权与下架权 |
| 平台承担优惠 | 平台承担的优惠券与积分抵扣不作为赛事方结算扣减项 |
| 结算详情展示 | 必须展示标价金额、优惠金额、用户净实付、平台承担金额等完整明细 |

### 8.9 合同、发票与审计

| 数据项 | 规则 |
|------|------|
| 合同对象 | 赛事方合作合同 / 渠道结算确认单 / 补充协议 |
| 合同状态 | 草稿 / 待签署 / 已签署 / 已失效 / 已归档 |
| 开票状态 | 待申请 / 审核中 / 开票中 / 已开票 / 已作废 / 已红冲 |
| 关联要求 | 合同、结算单、提现单、赛事方档案、渠道档案必须可关联 |
| 审计要求 | 审核、打款、补发、扣减、退款、回退均需留痕 |

### 8.10 动态表单规则

| 数据项 | 规则 |
|------|------|
| 表单驱动方式 | 赛事模板驱动 |
| 字段类型 | 文本 / 数字 / 单选 / 多选 / 日期 / 图片 / 音频 / 视频 / 文件 / 分组 / 条件字段 |
| 模板版本 | 变更后只影响新报名，不回写历史报名快照 |
| 存储模型 | 报名记录保存 `template_id + template_version + form_snapshot_json` 全量快照 |
| 校验责任 | 前端动态渲染 + 后端动态校验 |
| 文件限制 | 按字段配置类型、大小、数量 |
| 导出模型 | 后台按 `template_id / template_version` 解析 `form_snapshot_json` 动态生成 Excel 表头 |
| 禁用方案 | 不采用通用 EAV 作为主报名存储模型，避免查询与导出性能失控 |

### 8.11 风控与账务

- 风控对象覆盖：绑定、优惠券、积分、订单、退款、返点、售后、结算。
- 风险识别方式：规则阈值、频次阈值、人工标记、黑名单命中。
- 所有零付订单（零元订单 + 全额抵扣订单）均正常占用赛事名额、线下预约名额、商城库存，不因应付金额为 0 而绕过资源占用逻辑。
- 同用户 ID、同设备、同手机号、同渠道方对同一活动的零付下单频次必须接入风控中心限制与拦截。
- 账务要求：订单、积分、返点、结算、权益五类资产必须可追溯、可回滚、可审计。

### 8.12 线下预约与多码核销规则

| 数据项 | 规则 |
|------|------|
| 活动状态 | `preheat / booking / booked_out / ended / cancelled` |
| 预约粒度 | 当前版本按“日期 + 场次/时段”预约 |
| 重复预约 | 由后台按活动配置是否允许 |
| 预约类型 | `individual / team` |
| 团队预约资料 | 成员名单必填，并支持额外人数与额外名单（家长 / 工作人员） |
| 团队预约订单承载 | 付费团队预约不生成单笔团队总订单，而是按成员生成个人子订单并归集到团队主记录 |
| 额外人数口径 | 默认不生成个人子订单，仅记录预约扩展人数与名单 |
| 预约主状态 | `unbooked / booked / partial_writeoff / fully_writeoff / partial_writeoff_ended / cancelled / expired / no_show` |
| 核销子项类型 | `arrival / gift / ceremony / custom` |
| 核销子项状态 | `unused / used / expired / voided` |
| 预约与核销关系 | 1 个预约主记录可挂接 N 个核销子项 |
| 核销规则 | 每个子项独立二维码、独立状态、独立核销记录 |
| 核销人员 | 赛事方或平台授权现场人员 |
| 礼品库存联动 | 当前版本不联动库存 |
| 团队预约核销 | 支持整单核销或按人数部分核销 |
| 团队部分核销 | 允许跨多次累计完成，直到人数核销完毕或预约失效 |
| 场次结束终态 | 场次结束后：0 人核销记为 `no_show`；部分核销记为 `partial_writeoff_ended`；全部核销维持 `fully_writeoff` |
| 履约统计口径 | 团队预约相关现场履约、资源消耗、运营对账默认按实际已核销人数统计 |
| 财务收费口径 | 当前版本按预约锁定人数收费，不因部分未到场自动减免 |
| 未核销退款策略 | 场次结束后未核销人数默认不自动退款；如需补退，仅允许平台管理员后台人工部分退款 |
| 赛事方结算口径 | 与团队预约关联的有价成员子订单默认按预约锁定人数进入财务结算，除非合同明确约定按实际核销人数结算 |
| 扫码展示终端 | 固定使用 `event-writeoff.html`，并按个人 / 团队展示不同信息结构 |

## 9. 非功能与交付要求

### 9.1 非功能要求

- 首页首屏加载时间 <= 3 秒。
- 列表分页加载响应 <= 2 秒。
- 报名、积分扣减、返点计提、结算生成需保证幂等性。
- 个人信息需加密存储，证件信息需脱敏展示。
- 优先适配微信内置浏览器，同时支持 Safari / Chrome。
- 零元报名、实物兑换、售后回退、返点回退必须走完整状态流转。
- 渠道提现与赛事结算金额显示精度统一为两位小数。
- 平台 H5 审核端与赛事方 Web 工作台账号、Token、权限完全隔离。
- 横滑列表继续使用 `padding-left/right` 控制边距，禁止负 `margin`。
- 预约二维码、核销二维码必须支持一次性验真、幂等核销和核销日志留痕。

### 9.2 交付要求

- 外发时以本文档作为唯一主业务 PRD。
- 必须同时附带后台 Web PRD、状态机附录、数据模型附录、接口附录、页面验收附录和终端架构说明。
- 历史版本文档如需一并提供，必须明确标注“历史参考，不作为开发依据”。

## 10. 待确认事项

当前版本无高优先级待确认事项，以下口径已锁定为 AI 开发基线：

1. **临时参赛档案认领**
   - 当前版本默认按“监护人手机号 + 参赛人姓名”完全匹配自动认领。
   - 如仅手机号命中或存在多人共用手机号，由平台管理员人工确认、改绑或撤销误合并。
   - 赛事报名唯一编号仅作为辅助检索字段，不作为唯一认领键。

2. **团队预约资料**
   - 团队预约必须提交成员名称。
   - 同时支持填写额外人数与额外名单，用于家长、引路工作人员等非正式成员。

3. **团队部分核销**
   - 允许同一团队预约码跨多次累计核销。
   - 活动结束时，未核满团队自动流转为 `partial_writeoff_ended` 或 `no_show`。
