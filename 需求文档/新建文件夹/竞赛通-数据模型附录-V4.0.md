# 竞赛通-数据模型附录-V4.0

## 1. 文档定位

- 本附录用于补充 V4.0 的核心表设计、关键字段、主外键关系、唯一约束与金额口径。
- 本附录为对外开发交接所需的数据模型基线，实施团队可在不改变业务语义的前提下补充非关键字段。
- 本附录与《系统架构设计文档》《接口与第三方集成附录》《业务状态机与字典附录》共同构成数据与状态的单一事实源；数据表、字段、状态值、口径变更必须同步修订。

## 2. 建模原则

### 2.1 表命名与分层

1. 后台基础管理能力建议独立设计，不与业务账号和业务表混用。
2. 竞赛通业务表统一使用 `jst_*` 前缀。
3. 核心规则、台账、结算、风控、分析类表禁止混用 JSON 代替结构化字段。
4. 业务表按领域归属维护，至少明确：`channel / contest / trade / finance / asset / marketing / mall / partner / contractInvoice / risk / analysis / maic`。

### 2.2 通用字段规范

| 字段 | 类型建议 | 说明 |
|------|----------|------|
| `id` | `bigint` | 主键 |
| `biz_no` | `varchar(32)` | 业务单号，外显单号统一唯一 |
| `status` | `varchar(32)` | 当前状态，枚举值见状态机附录 |
| `remark` | `varchar(500)` | 备注 |
| `ext_json` | `json` | 非关键扩展字段，禁止承载核心计算字段 |
| `create_by` / `update_by` | `varchar(64)` | 操作人 |
| `create_time` / `update_time` | `datetime` | 创建更新时间 |
| `del_flag` | `char(1)` | 逻辑删除标记，关键财务表建议禁删只作废 |

### 2.3 金额与积分精度

| 项 | 类型建议 | 说明 |
|----|----------|------|
| 金额 | `decimal(12,2)` | 订单、退款、返点、服务费、发票金额 |
| 比例 | `decimal(8,4)` | 返点比例、折扣、服务费比例 |
| 积分 | `bigint` | 可消费积分 |
| 成长值 | `bigint` | 成长值不与消费积分混用 |
| 数量 | `int` | 库存、次数型权益 |

### 2.4 关键建模约束

1. `order_item` 为最小核算单元，退款、返点、服务费、结算、营销归因全部按明细落库。
2. 优惠券与积分分摊结果需落明细表，不允许只保留订单头汇总。
3. 所有财务类表必须具备来源单据号、关联单据号、正负方向、当前余额或变动值。
4. 经营分析报表来自交易与流水表聚合，不允许直接人工改写汇总结果。

### 2.5 DDL 与跨域变更约束

1. 所有 DDL 变更必须通过增量 SQL 或迁移脚本管理，禁止直接手改库结构后再补文档。
2. 涉及跨域表结构调整时，必须先确认主责领域，不允许在非主责需求中顺手修改别域核心表结构。
3. 财务、积分、成长值、权益、返点、提现、结算、库存、合同、发票、风控相关表禁止物理删除，只允许作废、冲正或逻辑失效。
4. 新增台账、流水、余额、状态字段时，必须同步补充索引策略、幂等策略、补偿策略和对账口径。
5. 破坏性字段变更必须提供兼容字段、数据迁移方案和回滚方案，不允许直接改字段语义。

补充说明：
- `jst_course.course_type` 当前最小口径统一为 `online/offline`，分别对应用户端“线上课程 / 线下班课”。

- 本轮“我的成绩/证书/课程/消息”页面收口未新增 DDL，仅补齐接口、字段说明和小程序页面骨架，不涉及测试库结构变更。

补充说明：
- `jst_points_account.points_balance` 存储账户总积分，`frozen_points` 存储冻结中的积分；用户端“可用积分”按 `points_balance - frozen_points` 计算，不再单独落字段。
- `jst_growth_account.current_level_id` 表示当前已生效等级；用户端等级进度通过 `jst_level_rule` 阈值区间实时计算，不允许在账户表冗余进度百分比。
- `jst_rights_template` 定义权益模板，`jst_level_rights_rel` 只表达“哪个等级会发什么权益”，真正已可用的权益实例统一以 `jst_rights_grant` 为准。
- `jst_rights_template.rights_type` 当前最小口径统一为 `enroll_deduction/venue_deduction/course_access/service_priority/manual_compensation`；历史值 `points_deduction/exclusive_course/venue_discount/service_privilege` 只作为迁移兼容来源，不再继续新增。
- `jst_rights_grant.total_quota` 表示发放时总配额，`available_quota` 表示当前剩余可用配额；金额型、次数型统一使用 `DECIMAL(12,2)`，由 `quota_type` 决定业务解释。
- `jst_rights_writeoff` 同时承担“权益核销申请单”和“核销结果单”职责，审核、确认、回退字段全部落主表；所有操作痕迹另落 `jst_rights_writeoff_audit_log`。
- `jst_rights_ledger` 只记录真正影响权益余额的台账动作，当前最小 `change_type` 口径为 `grant/writeoff_use/writeoff_reverse/expire`。

## 5. 关键关系与约束

### 5.1 用户与绑定

- `jst_user` 与 `jst_student_profile` 一对一或一对多，视一个用户是否管理多个学生档案而定。
- `jst_user.mobile` 在首次微信登录后允许为空，直到完成手机号绑定；`mobile_bind_status=unbound/bound` 用于约束“未绑手机号仅可浏览”的前端与交易行为。
- `jst_user_wechat_auth` 负责保存 `openid/unionid/session_key` 映射，业务账号主表与第三方账号映射表分离。
- `jst_channel.owner_user_id` 用于确认当前登录用户是否具备渠道方个人中心、认证、统计、代报名等渠道视角能力；学生“绑定渠道”和“自有渠道”必须分离建模，禁止继续混用。
- `jst_student_bind` 保存全部历史绑定记录，其中仅允许一条有效记录。
- 历史订单、返点台账按当时支付渠道归属，不随绑定变更回写。

### 5.2 订单与分摊

- `jst_order` 保存汇总金额，`jst_order_item` 保存最小核算单元。
- `jst_order.owner_type + owner_biz_id` 保存当前订单归属，用于后续返点、结算、订单详情展示，不随绑定变更回写历史订单。
- 优惠券、积分、平台服务费均需按 `order_item` 保存分摊结果。
- `jst_payment_record.pay_params_json` 保存支付参数快照，用于支付重放、问题排查与审计留痕。
- `jst_transfer_voucher.file_url` 作为订单详情默认主预览图，多张转账凭证文件明细统一落 `jst_transfer_voucher_file`。
- `jst_transfer_voucher.review_by/review_by_name/review_time` 记录最近一次财务审核结果，用于用户端订单详情和 H5 审核端留痕回显。
- `jst_transfer_voucher_audit_log` 保留每次审核动作和审核附件，不能被 `jst_transfer_voucher` 主记录覆盖。
- `jst_callback_log` 记录 `/api/open/**` 回调原始报文、签名字段、处理结果与重复回调次数，是回调幂等与审计留痕事实源。
- `jst_async_task` 承接 `close_order`、支付补单、通知重试等异步补偿动作；本期至少落地支付超时关单与 mock 支付回调任务。
- 当前测试库中仍兼容保留历史列 `bank_name/transfer_time/state/reject_reason/auditor_id/audit_time`；这些列只用于迁移兼容和历史回填，不作为 V4 新接口事实源。
- 部分退款时只回滚被退款明细上的分摊金额，不回滚未退款明细。

### 5.3 返点与服务费

- `jst_rebate_rule` 仅按 `contest_id + channel_id` 生效。
- `jst_service_fee_rule` 支持平台默认、赛事级、赛事 + 渠道级，需要通过 `scope_type` 区分。
- `jst_rebate_ledger` 必须区分正向台账与负向抵扣台账，推荐字段 `ledger_type=positive/negative/offset`。

### 5.4 权益与等级

- 等级是规则，权益是发放实例，不能直接在等级表上扣减配额。
- `jst_level_rights_rel` 负责等级与权益模板的映射，真正可用权益以 `jst_rights_grant` 为准。
- 核销、回退、过期一律落 `jst_rights_ledger`。

### 5.5 合同与开票

- 合同、开票必须关联提现单、赛事结算单或其他业务单据，不允许孤立存在。
- 同一业务单号可以对应多张发票记录，但必须有金额累计控制。

## 6. 开发落地说明

1. 后台菜单、角色、按钮权限通过独立后台权限模型管理。
2. 业务账号和平台账号不得共用同一登录表。
3. 平台管理 Web 与平台 H5 审核端可共用后台账号主表，但必须使用独立会话上下文、审计日志和终端标识。
4. 所有财务台账类表必须禁用物理删除，只允许作废或冲正。
5. 发生跨表状态流转时，必须保证“状态机附录、接口附录、数据模型附录”三者一致后再开发。
6. 经营分析汇总表建议按日离线汇总，保留从交易表全量重算能力。
