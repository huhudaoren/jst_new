# 竞赛通-业务状态机与字典附录-V4.0

## 1. 文档定位

- 本附录用于补充《竞赛通-产品需求文档-V4.0》与《竞赛通-管理后台Web端需求文档-V4.0》的状态流转、枚举值和审核边界。
- 本附录是 V4.0 对外开发交接包的正式组成部分，状态机、字典、流转规则均以本附录为准。
- 如主文档与本附录存在冲突，以本附录的状态定义和资金流转定义为准，再回写主文档修订。

## 2. 全局设计原则

1. 最小核算单元为 `order_item` 或 `enroll_item`。优惠券、积分、平台服务费、返点、退款、赛事结算均按最小核算单元分摊和回滚。
2. 单据状态与资金状态分离。比如订单已支付，不等于报名已审核通过；提现已审核通过，不等于已打款。
3. 平台管理 Web 与平台 H5 审核端共用同一套后端状态机，不允许出现两套独立状态。是否可执行终审，由角色权限控制。
4. 所有审核、打款、核销、发货、红冲、回退动作必须记录操作人、操作时间、原因、附件。
5. 所有外部回调必须幂等，同一业务流水号重复回调不能重复变更状态。
6. 除特殊说明外，列表页只展示“当前有效状态”，历史流转通过时间轴或流水表追溯。

### 4.6 返点台账

| 当前状态 | 可流转到 | 触发动作 | 操作角色 | 说明 |
|----------|----------|----------|----------|------|
| `pending` | `available/reversed` | 赛事结束后计提；如已退款则回退 | 系统 | 仅针对已支付且未全退明细 |
| `available` | `reviewing/reversed` | 渠道发起提现或退款回退 | 渠道方 / 系统 | 可提现余额来源于该状态 |
| `reviewing` | `paid/rejected/reversed` | 提现审核通过打款、驳回、退款回退 | 财务员 / 系统 | 驳回后回到 `available` |
| `paid` | - | 已打款完成 | 财务员 | 若后续发生退款，新增负向抵扣台账 |
| `rejected` | `available` | 提现驳回后释放金额 | 财务员 | 仅表示本次提现审核被驳回 |
| `reversed` | - | 回退结束 | 系统 / 财务 | 未打款回退直接冲减原台账 |

补充规则：

- 固定金额返点按单个 `order_item` 计提，不按整单计提。
- 返点规则只支持“赛事 + 渠道方”维度；未配置视为不返点。
- 平台服务费从返点基数中扣减，公式为：`返点基数 = max(0, 净实付 - 平台服务费)`。

### 4.8 优惠券

| 当前状态 | 可流转到 | 触发动作 | 操作角色 | 说明 |
|----------|----------|----------|----------|------|
| `unused` | `locked/expired/voided` | 下单锁券、到期失效、人工作废 | 系统 / 运营 | 单笔订单默认仅使用 1 张优惠券 |
| `locked` | `used/unused/voided` | 支付成功、取消订单解锁、后台作废 | 系统 / 运营 | 锁券需设置超时释放 |
| `used` | `unused/voided` | 全额退款恢复、异常作废 | 系统 / 财务 | 仅整单全退且仍在有效期内可恢复 |
| `expired` | - | 到期结束 | 系统 | 不可恢复 |
| `voided` | - | 作废结束 | 运营 / 风控 | 需记录作废原因 |

补充规则：

- 优惠券优先于积分抵扣。
- 订单部分退款时，不恢复原优惠券，仅回滚被退款明细分摊的优惠金额到财务核算。
- 整单全退时，若券模板配置允许退款恢复且券未过期，则恢复为 `unused`，保留一条回退流水。

## 5. 补充约束

### 5.1 审核边界

- PC 后台承担规则配置、最终财务审核、发票处理、合同归档、风险规则配置。
- 平台 H5 审核端仅承接高频审核、现场确认、快速标记动作，不承担复杂规则配置。
- 若平台 H5 审核端被赋予与平台管理 Web 相同的审核权限，则必须复用同一接口、同一状态机、同一流水表。

### 5.2 资金与资产回滚顺序

1. 退款时先回滚 `order_item` 级优惠金额与积分金额。
2. 再回滚平台服务费、返点计提、赛事结算分摊金额。
3. 如权益已核销，需先完成权益回退或扣减应退金额，再进入现金退款。
4. 如返点已打款，则新增负向台账，后续提现或结算时自动抵扣。

### 5.3 优先级规则

- 返点规则优先级：仅支持 `赛事 + 渠道方`，未配置即不返点。
- 平台服务费优先级：`赛事 + 渠道方` > `赛事` > `平台默认`。
- 合同优先级：同一业务对象仅允许一份 `signed` 主合同生效，补充协议需单独编号并挂靠主合同。

---


### 5.4 订单关闭补充规则

- `pending_pay -> closed`：允许由系统超时关单触发，也允许由用户主动取消触发。
- `rejected -> closed`：允许用户在转账驳回后主动取消订单。
- `paying/transfer_reviewing -> closed`：仅允许系统补偿任务或风控/财务链路触发，前端不提供主动关闭入口。
- 赛事订单用户主动取消后：`jst_order.pay_status=closed`，报名单状态写为 `cancelled`。
- 赛事订单支付超时关闭后：`jst_order.pay_status=closed`，报名单状态写为 `closed`。
- 课程订单关闭后：仅变更订单状态，不联动报名状态。
- `close_reason` 必须同步写入，禁止只改状态不留原因。
