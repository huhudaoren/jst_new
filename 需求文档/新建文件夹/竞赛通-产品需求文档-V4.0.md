## 2. 项目背景

竞赛通定位为面向学生、家长、渠道方、赛事方及平台管理员的一站式竞赛服务平台，当前以文化艺术类赛事为核心赛道，提供“找赛事、报赛事、查成绩、看证书、学课程”的完整服务链路。

历史方案虽已覆盖基础报名、课程与审核能力，但作为重新开发的 V4.0 版本，需一次性补齐以下业务短板：

1. 缺少渠道激励体系。学生绑定老师或机构后，平台无法基于报名订单为对应渠道方计算返点，难以持续激励招生与代报名。
2. 缺少积分与等级体系。平台只能承接一次性报名交易，用户留存、复购与权益运营能力不足。
3. 缺少赛事供给侧平台化能力。当前赛事主要由平台自营或人工录入，缺少第三方赛事公司自助入驻、发赛、管赛、结算的标准流程。

**V4.0 新增方向：**
- **渠道返点与结算体系**：学生可绑定渠道方（老师 / 机构 / 个人），订单按“在哪个渠道支付就算哪个渠道”归属，赛事结束后按规则结算返点，支持提现申请、平台审核、打款状态、结算单、发票信息。
- **积分等级与积分商城体系**：支持成长值决定等级、消费积分用于兑换，积分来源覆盖报名、购课、签到、邀请、学习完成、获奖等行为；等级权益按角色灵活配置，支持报名抵扣、场地减免、专属课程、客服特权等；积分商城支持虚拟商品与实物商品。
- **赛事方入驻体系**：支持第三方赛事公司申请入驻，审核通过后进入独立工作台，自助发布赛事、管理报名、导入成绩、发放证书，并由平台保留最终审核权与下架权。
- **优惠券 / 营销中心**：支持优惠券模板、发券、活动专题、报名优惠组合与营销活动运营，形成“积分 + 优惠券 + 活动”联合促转化能力。
- **权益核销中心**：支持等级权益、活动权益、人工补发权益的发放、申请、核销、回退与台账对账，重点覆盖场地减免、专属课程、客服特权等权益闭环。
- **经营分析中心**：支持用户、渠道、赛事、营销、AI 课程、财务等多维经营分析，形成统一经营驾驶舱。
- **风控中心**：支持绑定异常、刷积分、刷券、恶意退款、异常返点、异常发货等风险识别、预警、处置与复核。
- **合同与开票自动化**：支持赛事方合作合同、渠道结算单、发票申请、开票状态、打款凭证与结算对账的标准化流转。

## 3. 产品目标

### 3.1 业务目标
- 建立统一的文化艺术赛事信息展示与报名入口，持续提升报名转化率。
- 沉淀用户、订单、成绩、证书、课程、积分、返点、结算等核心数据资产。
- 建立渠道激励机制，提升渠道招募能力与代报名规模。
- 建立积分与等级运营体系，提升用户活跃、复购与长期留存。
- 建立赛事方入驻能力，扩大赛事供给，形成平台化交易生态。
- 建立营销与权益体系，提升拉新、促活、转化与复购。
- 建立经营分析与风控能力，提升平台经营效率与资金安全。
- 建立合同与开票自动化能力，降低合作与结算的人工作业成本。
- 保留 AI 定制课程差异化优势，强化“赛事 + 学习 + 运营”闭环。

### 3.2 用户目标
- 学生 / 家长：找赛事、报名缴费、赚积分、用积分抵扣或兑换商品、查成绩、下载证书、学习课程。
- 学生 / 家长：找赛事、报名缴费、领券抵扣、赚积分、兑换商品、使用权益、查成绩、下载证书、学习课程。
- 渠道方：绑定学生、代报名、统一支付、查看数据、查看返点、申请提现、使用等级权益与核销权益。
- 赛事方：申请入驻、发布赛事、管理报名、导入成绩、管理证书、查看合同、发起开票、查看结算。
- 管理员：在移动端审核高频任务，在后台完成返点、积分、营销、权益、赛事方、风控、合同与结算管理。

### 3.3 V4.0 阶段目标
- 完成竞赛通从 0 到 1 的重新设计与重新开发基线定义。
- 覆盖完整的报名、课程、审核、运营、财务与结算基础能力。
- 完成渠道返点“规则配置 -> 订单归属 -> 自动计提 -> 提现申请 -> 审核打款”闭环。
- 完成积分“获取 -> 成长升级 -> 权益生效 -> 积分抵扣 -> 商城兑换 -> 售后回滚”闭环。
- 完成优惠券“模板配置 -> 发券 -> 下单核销 -> 退款回滚 -> 活动分析”闭环。
- 完成权益“权益配置 -> 发放 -> 申请 -> 核销 -> 回退 -> 对账”闭环。
- 完成赛事方“入驻申请 -> 赛事发布 -> 平台审核 -> 报名管理 -> 成绩导入 -> 发证 -> 财务结算”闭环。
- 完成合同与开票“合同生成 -> 签署确认 -> 开票申请 -> 审核开票 -> 结算打款”闭环。
- 完成风控“规则配置 -> 风险识别 -> 预警 -> 处置 -> 复核”闭环。
- 完成经营分析“经营总览 -> 主题分析 -> 异常预警 -> 报表导出”闭环。
- 完成角色模型纠偏：取消“机构管理多个老师”的错误层级，统一为渠道方角色模型。

## 4. 用户角色

### 4.1 C 端用户
- **学生**：浏览赛事、自主报名、查看进度、查询成绩、下载证书、赚取积分、参与积分兑换、学习课程与 AI 课程。
- **家长**：协助学生完成报名、支付、资料上传、地址填写、兑换跟踪与售后沟通。

### 4.2 渠道方

> 渠道方为统一业务角色，不再区分“机构管理多个老师”的层级结构。老师、机构、个人只是渠道方的不同认证类型，不存在上下级关系。

- **老师渠道方**：绑定学生、代报名、统一支付、查看数据、查看返点、申请提现、使用等级权益。
- **机构渠道方**：业务能力与老师渠道方一致，仅认证材料与展示名称不同。
- **个人渠道方**：适用于独立招生个人，具备绑定学生、代报名、返点结算、积分权益等能力。

### 4.3 赛事方
- **赛事方**：入驻的第三方赛事公司，审核通过后可在独立工作台中管理自身赛事、报名、成绩、证书和结算。

### 4.4 平台方
- **平台管理员**：负责用户审核、报名审核、凭证审核、赛事方入驻审核、渠道提现审核、赛事结算审核、公告发布、成绩发布、AI 课程审核等。

### 4.5 角色关系说明
- 学生与渠道方为**一对一有效绑定关系**，同一时间仅能绑定一个渠道方。
- 学生与赛事方**不建立绑定关系**，学生只是报名赛事方发布的赛事。
- 历史订单归属遵循“**在哪个渠道支付就算哪个渠道**”，学生后续更换渠道方不影响历史订单返点归属。
- 赛事方不参与积分等级体系；积分等级仅面向学生与渠道方。

## 5. 使用场景

1. 学生在首页浏览赛事，选择合适赛事报名，使用优惠券与积分抵扣后支付，支付成功获得积分与成长值。
2. 学生扫描渠道方绑定码，与某老师 / 机构 / 个人渠道方建立一对一绑定关系，由渠道方统一代报名与支付。
3. 渠道方为多个学生代报名不同赛事，赛事结束后自动生成返点明细，渠道方在返点中心申请提现。
4. 学生使用积分直接抵扣报名金额，若订单金额为 0，则系统自动完成零元报名。
5. 渠道方因等级提升获得场地减免权益，在渠道中心查看可用权益并申请线下核销。
6. 用户在积分商城兑换实物商品，填写收货地址，跟踪发货状态和物流单号，并在需要时发起售后 / 退款。
7. 渠道方因等级提升获得场地减免权益，在权益中心发起核销申请，平台或合作方完成核销确认后生成权益台账。
8. 第三方赛事公司提交入驻申请，经平台审核通过后，登录赛事方工作台发布新赛事。
9. 赛事方在线查看合作合同、确认合同条款、提交开票资料，并在赛后查看结算与打款状态。
10. 平台根据合同配置的平台服务费、渠道返点和其他约定项，生成赛事方结算单并完成打款。
11. 平台运营通过优惠券与营销活动中心配置活动专题，观察报名转化、用券率、复购率等经营指标。
12. 平台风控识别异常绑定、异常退款、异常积分或异常返点，生成风险工单后交由运营 / 财务复核。
13. 老师上传教学材料生成 OpenMAIC AI 互动课件，学生在 AI 课堂中学习，学习完成后同步进度并获得对应积分 / 成长值。

## 6. 产品范围

### 6.1 产品定位
竞赛通 V4.0 定位为以微信小程序为核心业务端、同时支持赛事方独立 Web 工作台的竞赛服务平台，核心业务闭环为：

`找赛事 -> 绑定渠道 -> 报名/代报名 -> 优惠券/积分抵扣 -> 支付 -> 审核 -> 成绩/证书 -> 积分/等级 -> 商城兑换 / 权益核销 -> 返点/结算 / 合同开票 -> AI 学习 / 经营分析`

- 本主 PRD 覆盖微信小程序端、赛事方公开申请页、赛事方 Web 工作台及平台 H5 审核端。

- 配套终端拆分、账号体系、部署边界与技术架构建议，见《竞赛通-终端形态与架构规划说明-V4.0》。

## 7. 核心流程

### 7.1 学生自主报名与积分获得流程
```

 -> 赛事列表 -> 赛事详情 -> 登录
 -> 报名流程（填资料 -> 选类别 -> 上传作品 -> 支付）

### 7.2 渠道绑定、代报名与返点流程
```

 -> 学生带参直达 binding.html
 -> 或学生从个人中心主动进入 binding.html 手动输入绑定码兜底

### 9.2 登录入口（login.html / event-login.html / admin-login.html）

**用户端登录（login.html）**
- 微信小程序内默认采用微信授权登录，首次登录需完成手机号绑定。
- 未完成手机号绑定时，仅允许浏览，不允许进行报名支付、积分兑换、提现申请等需实名联系方式的操作。
- 登录后可根据认证状态切换学生视角与渠道方视角。
- 小程序端用户登录页不保留显性的“赛事方登录 / 管理员登录”跨端入口。
- 赛事方登录与管理员登录通过各自专用访问路径进入，不作为小程序用户登录主流程的一部分。

**赛事方登录（event-login.html）**
- 账号密码 + 图形验证码登录。
- 仅面向已审核通过的赛事方账号。
- 首次登录强制修改初始密码。
- 未通过审核的赛事方不开放正式工作台，仅支持在公开申请页查看审核进度、补充资料和重新提交。
- 支持“去入驻申请”跳转 `event-apply.html`。
- 登录成功进入 `event-home.html`。

**管理员登录（admin-login.html）**
- 账号密码 + 图形验证码 + 连续失败锁定机制。

**赛事详情**

**报名流程**

### 9.6 返点与结算模块（V4.0 新增）

#### 核心规则
- 返点按“**赛事 + 渠道方**”单独配置。
- 支持两种返点模式：比例返点、固定金额返点。
- 返点默认公式：`返点金额 = max(0, 净实付 - 平台服务费) * 返点比例` 或固定金额。
- 净实付定义：`订单支付金额 - 优惠券抵扣 - 积分抵扣`。
- 平台服务费默认参与扣减，且支持后台灵活配置。
- 资金核算以 `order_item` 为最小单元，优惠券、积分、服务费、返点、退款按明细分摊与回滚。
- 返点结算时点：赛事结束后，且订单未发生退款。
- 历史订单归属不因学生更换渠道方而回溯调整。
- 退款规则：已计提但未打款的返点原路扣回；已打款返点进入负向台账，后续结算中抵扣。

#### channel-rebate.html — 返点中心
- Hero：可提现余额 / 审核中金额 / 累计已打款 / 历史退款回退。
- 状态 Tab：待结算 / 可提现 / 审核中 / 已打款 / 已驳回 / 已回退。
- 明细字段：赛事名称、学生姓名、订单号、净实付、平台服务费、返点规则、返点金额、赛事结束时间。
- 操作：查看明细、勾选提现、导出结算单。

#### channel-settlement.html — 提现与结算单
- 状态 Tab：待提交 / 审核中 / 已撤回 / 已驳回 / 待打款 / 已打款。
- 提现申请：选择结算项、填写发票信息、确认收款账户。
- 结算单：结算编号、申请时间、审核状态、打款状态、打款时间、驳回原因。
- 支持查看历史发票抬头、收款账户、打款凭证。
- 作为渠道方财务结算主入口，提供跳转 `contract-center.html` 与 `invoice-center.html` 的入口。

### 9.7 积分与等级模块（V4.0 新增）

#### 核心规则
- 系统同时维护两套资产：
  - **成长值**：决定等级，默认只增不减，后台可人工调整。

  - 固定值奖励

  - 按角色差异化奖励

### 9.20 风控与异常处理

#### 核心规则

- 风险工单必须支持创建、分派、复核、关闭、回溯。

#### 典型风险场景
- 同设备 / 同手机号高频切换渠道绑定。
- 异常集中使用优惠券、短时间内大量报名后退款。
- 同用户 / 同渠道出现异常积分增长或异常返点增长。
- 商城实物异常退货、虚假物流、恶意售后。

幼儿园组 / 小学低年级组 / 小学高年级组 / 初中组 / 高中组

### 10.5 学生与渠道方关系规范
| 数据项 | 规则 |
|--------|------|
| 有效绑定数 | 同一时间仅 1 条有效绑定 |
| 绑定对象 | 渠道方（老师 / 机构 / 个人） |
| 更换绑定 | 允许更换，新绑定生效后旧绑定失效 |
| 历史订单归属 | 以支付发生时的渠道方为准，不因后续更换而变更 |
| 赛事方关系 | 学生不与赛事方绑定 |

### 10.6 返点与服务费规则
| 数据项 | 规则 |
|--------|------|
| 配置颗粒度 | `赛事 + 渠道方` |
| 返点模式 | `rate`（比例）/ `fixed`（固定金额） |
| 平台服务费 | 支持按比例 / 固定金额 / 不收取，后台可配置 |
| 默认公式 | `返点基数 = max(0, 净实付 - 平台服务费)` |
| 净实付 | `支付金额 - 优惠券抵扣 - 积分抵扣` |
| 结算时点 | 赛事结束且订单未退款 |
| 提现状态 | `pending` / `reviewing` / `rejected` / `approved` / `paid` |
| 退款回退 | 已计提返点原路扣回；已打款转负向台账 |

### 10.7 积分与成长值规则
| 数据项 | 规则 |
|--------|------|
| 参与角色 | `student` / `channel` |
| 赛事方 | 不参与积分等级 |
| 成长值 | 决定等级，默认不失效 |
| 积分 | 用于抵扣、兑换、售后回滚 |
| 抵扣比例 | `100 积分 = 1 元` |
| 过期规则 | 永不过期 |

## 11. 非功能需求

- 首页首屏加载时间 <= 3 秒。
- 列表分页加载响应 <= 2 秒。
- 报名、积分扣减、返点计提、结算生成需保证幂等性。
- 订单、积分、返点、结算四类账务需可追溯、可回滚、可审计。
- 个人信息加密存储，证件信息脱敏展示。
- 优先适配微信内置浏览器，同时支持 Safari / Chrome。
- 零元报名订单需支持正常创建、审核和通知流程。
- 实物兑换订单需支持物流单号记录与售后状态流转。
- 渠道提现与赛事结算页面的金额展示精度统一为两位小数。
- 优惠券、积分、权益、返点、结算五类资产需保证账务可追溯。
- 风控预警、合同签署、开票流转、报表导出需保留操作痕迹。
- 横滑列表继续使用 `padding-left/right` 控制边距，禁止负 margin。
- 平台 H5 审核端与赛事方 Web 工作台账号、Token、权限完全隔离。

## 12. 后台能力范围（对应配套后台 PRD）

### 12.1 基础能力
- 赛事管理、报名管理、订单管理、内容管理、成绩管理、证书管理、用户认证管理、AI 课程管理。

### 12.2 V4.0 新增后台能力
- **渠道方管理**：渠道认证、绑定码、渠道关系、渠道统计。
- **返点规则管理**：按“赛事 + 渠道方”配置返点模式、返点值、平台服务费规则。
- **渠道结算管理**：提现审核、打款记录、结算单、发票信息。
- **积分规则引擎**：行为规则、积分发放、成长值发放、角色差异化配置。
- **等级与权益管理**：等级数量、名称、图标、阈值、角色权益配置。
- **优惠券与营销管理**：优惠券模板、发放、活动专题、营销活动、营销分析。
- **权益核销管理**：权益模板、权益发放、核销申请、核销记录、权益台账。
- **积分商城管理**：商品、库存、上下架、订单、物流、售后。
- **赛事方管理**：入驻申请、赛事审核、赛事上下架、合同约定项、结算规则。
- **合同与开票管理**：合同模板、签署状态、开票申请、开票记录、票据归档。
- **赛事结算管理**：赛事方结算单、打款状态、发票管理。
- **风控中心**：风控规则、风险预警、风险工单、黑名单、异常审计。
- **经营分析中心**：渠道分析、赛事分析、营销分析、财务分析、AI 课程分析。
- **消息模板管理**：渠道提现、赛事结算、积分变更、兑换物流、售后结果等通知模板。

### 12.3 H5 管理员与 PC 后台边界
- H5 管理员聚焦：审核、发布、确认、打款状态标记。
- PC 后台聚焦：复杂配置、规则维护、合同维护、批量导入导出、对账与报表。
