# 竞赛通-终端形态与架构规划说明-V4.0

## 1. 文档定位

- 本文档用于明确竞赛通 V4.0 的终端形态、账号边界、技术架构建议与对外交付边界。
- 本文档与《竞赛通-产品需求文档-V4.0》《竞赛通-管理后台Web端需求文档-V4.0》共同构成 V4.0 对外重新开发的规划基线。

## 2. 终端形态结论

### 2.1 正式终端方案

| 终端 | 主要使用角色 | 形态 | 说明 |
|------|--------------|------|------|
| 业务端 | 学生 / 家长 / 渠道方 | 微信小程序 | 平台核心业务端，承接报名、支付、积分、权益、渠道工作台 |
| 赛事方申请端 | 待入驻赛事方 | 公开申请页 | 用于提交入驻资料、查看审核进度、补充材料 |
| 赛事方工作台 | 已审核通过赛事方 | Web | 用于创建赛事、管理报名、导入成绩、发证、结算、合同开票 |
| 平台管理端 | 平台管理员、运营、财务、风控 | 平台管理 Web | 承接复杂配置、规则维护、审核、对账、报表 |
| 平台移动审核端 | 平台审核员 | H5 审核端 | 仅承接高频审核与现场确认，不承接复杂配置 |

### 2.2 为什么此时必须定终端

- 微信小程序与 H5 的登录、支付、分享、消息、上传、文件预览、客服能力不同。
- 赛事方 B 端场景包含赛事编辑、Excel 导入、批量发证、合同、开票、结算，天然更适合 Web。
- 若先做原型再决定终端，信息架构、交互方式、接口设计、技术选型都会返工。

结论：

- 终端形态必须在 PRD 阶段先定，不应等原型完成后再决定。

## 3. 各终端功能边界

### 3.1 微信小程序

承接能力：

- 赛事浏览、报名、资料上传、支付、退款申请
- 个人中心、绑定渠道、我的订单、成绩、证书
- 积分中心、等级权益、优惠券、权益核销、积分商城、售后
- 渠道方认证、学生管理、渠道订单、返点中心、提现申请、数据分析
- AI 课程入口、营销专题、分享拉新、订阅消息

不承接能力：

- 平台复杂规则配置
- 赛事方正式工作台
- 大批量 Excel 导入导出
- 合同模板维护、发票后台审核、风控规则配置

### 3.2 赛事方公开申请页

承接能力：

- 提交入驻申请
- 保存草稿
- 查看审核状态
- 按驳回意见补充资料并重提

不承接能力：

- 正式赛事管理
- 报名管理
- 成绩导入
- 证书管理
- 结算与合同开票

### 3.3 赛事方 Web 工作台

开通条件：

- 赛事方入驻申请审核通过
- 平台创建或激活赛事方账号
- 赛事方完成首次登录与密码初始化

承接能力：

- 赛事创建、编辑、提审、上下架协同
- 报名管理、数据导出
- 成绩导入、证书管理、发布申请
- 赛事结算确认、合同查看、开票申请
- 工作台消息、资料维护、子账号协作

### 3.4 平台管理 Web

承接能力：

- 全部复杂配置能力
- 渠道、赛事方、赛事、订单、积分、商城、营销、权益、风控、财务、合同、开票、分析全量后台能力
- 权限、菜单、角色、数据权限、日志审计

技术基线：

- 采用独立前后端分离管理后台架构
- 前端：独立管理后台前端工程
- 后端：管理后台服务 + 业务服务

### 3.5 平台 H5 审核端

承接能力：

- 高频审核
- 现场确认
- 标记已打款
- 风险确认和转交

不承接能力：

- 复杂规则配置
- 大批量管理
- 对账报表
- 合同模板与营销模板维护

## 4. 账号与登录规划

### 4.1 账号体系

| 账号体系 | 角色 | 登录方式 |
|----------|------|----------|
| 小程序业务账号 | 学生 / 家长 / 渠道方 | 微信授权登录 + 手机号绑定 |
| 赛事方账号 | 赛事方主账号 / 子账号 | Web 账号密码登录 |
| 平台账号 | 平台管理员 / 运营 / 财务 / 风控 | 平台管理后台账号密码登录 |
| H5 审核账号 | 平台审核员 | 独立审核账号或复用平台账号 Token 体系 |

### 4.2 关键规则

1. 学生 / 家长 / 渠道方共用一套小程序账号体系，通过身份与认证状态切换页面。
2. 赛事方必须“先申请、后审核、再开通 Web 账号”。
3. 赛事方审核未通过前，不开放正式工作台。
4. 平台管理账号、H5 审核账号、赛事方账号与小程序业务账号必须逻辑隔离。

## 5. 技术架构建议

### 5.1 总体架构

```text
微信小程序
    |
    | HTTPS API
    v
业务后端（Java / Spring Boot 或同类企业级框架）
    |
    +-- 平台管理后台 Web
    +-- 赛事方 Web 工作台
    +-- 平台 H5 审核端
    +-- MySQL / Redis / OSS
    +-- 微信支付 / 短信 / 物流 / 电子签章 / 开票平台 / OpenMAIC
```

### 5.2 前端拆分建议

| 前端工程 | 建议 |
|----------|------|
| 微信小程序端 | 独立工程，承接学生 / 家长 / 渠道方 |
| 赛事方 Web 工作台 | 独立 B 端工程，面向赛事方 |
| 平台管理 Web | 独立管理后台前端工程 |
| 平台 H5 审核端 | 轻量移动端工程，可独立部署 |

### 5.3 后端拆分建议

| 模块 | 说明 |
|------|------|
| 用户与认证 | 小程序登录、渠道认证、赛事方账号、平台账号 |
| 赛事中心 | 赛事方、赛事、报名、成绩、证书 |
| 交易中心 | 订单、支付、退款、对账 |
| 资产中心 | 积分、成长值、优惠券、权益、返点 |
| 商城中心 | 商品、库存、兑换、物流、售后 |
| 财务中心 | 服务费、提现、赛事结算、打款、开票 |
| 风控中心 | 风控规则、预警、工单、黑白名单 |
| 内容与 AI | 公告、专题、OpenMAIC 接入 |
| 分析中心 | 经营总览、渠道、赛事、营销、财务分析 |

## 6. 小程序专项架构约束

### 6.1 必须按小程序能力设计的点

- 微信授权登录、手机号绑定
- 微信支付拉起
- 订阅消息通知
- 小程序分享链路与邀请场景
- 图片上传、文件预览、地址管理、客服接入

### 6.2 需要特殊方案的点

| 能力 | 建议方案 |
|------|----------|
| OpenMAIC AI 课堂 | 优先评估小程序原生承载；复杂交互场景可通过 `web-view + SSO` 方案接入 |
| 合同查看与签署 | 小程序仅做查看和跳转，正式签署建议跳 Web 或第三方签章页面 |
| 证书下载 | 小程序内预览或临时文件下载，复杂格式走 OSS 临时链接 |
| 赛事方申请页 | 可保留公开 H5 申请页，并由小程序或官网引流进入 |

## 7. 对外交付建议

### 7.1 应发送给外部团队的文档包

必须发送：

- `V4.0` 主 PRD
- `V4.0` 管理后台 Web PRD
- 4 份开发附录
- 本《终端形态与架构规划说明-V4.0》

默认不发送：

- `V3.0` 及以前版本

### 7.2 关于是否发送 V3.0

建议：

- 如果是正式重新开发，不需要发送 `V3.0`。
- 因为 `V4.0` 已经是独立交付版，且角色模型、终端形态、返点、积分、赛事方入驻等核心内容都已经重写。
- 如果把 `V3.0` 和 `V4.0` 一起发给外部团队，极易造成“按旧逻辑实现”的偏差。

仅在以下场景才可附带 `V3.0`：

- 对方明确需要了解历史演进
- 你在邮件或交接说明中明确标注：`V3.0 仅供历史参考，不作为开发依据`

## 8. 结论

- 竞赛通 V4.0 的正式终端方案应固定为：`微信小程序 + 赛事方 Web + 平台管理 Web + 平台 H5 审核端 + 赛事方公开申请页`。
- 该终端方案应作为原型设计、技术架构设计、排期评估和开发实施的前置条件。
- 对外重新开发时，以 `V4.0` 文档包为唯一基线，不以 `V3.0` 为开发依据。
